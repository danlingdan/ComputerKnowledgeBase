# 3.1 数据链路层的基本问题

数据链路层把网络层交下来的数据封装成帧，通过一段链路交给相邻节点。不同链路协议形式各异，但都要回答：怎样识别帧边界、怎样允许任意数据通过，以及怎样发现传输错误。

## 链路、数据链路与帧

- **链路（link）**：相邻节点之间没有其他交换节点的一段物理通路；
- **数据链路（data link）**：物理链路加上控制数据传输所需的协议及实现；
- **帧（frame）**：数据链路层的协议数据单元，通常由首部、载荷和尾部组成。

网络适配器通常同时实现数据链路层和物理层的部分功能，因此软硬件边界不必与概念分层完全重合。

![[Pasted image 20260715231953.png]]

> [!example] 相邻节点交付
> 节点 A 把 IP 数据报封装成帧并发送；节点 B 检查帧，若通过检错就取出 IP 数据报交给网络层，否则丢弃。中间物理信号怎样表示比特，由[[第二章 物理层|物理层]]处理。

## 基本问题一：封装成帧

封装成帧（framing）在数据前后加入控制信息，使接收端能从连续比特流中识别帧边界：

```mermaid
flowchart LR
    H["帧首部"] --> D["网络层数据 / 载荷"] --> T["帧尾部"]
```

![[Pasted image 20260715232002.png]]

帧首尾还可以携带地址、协议类型和差错检测信息。链路协议通常规定最大传送单元（Maximum Transmission Unit, MTU）：

> [!definition] MTU
> MTU 是帧中可承载的上层数据部分的最大长度，不是整个帧的最大长度。首部、尾部以及物理层前导信息通常不计入 MTU。

较大载荷可降低首尾开销比例，但也会增加单帧发送时间、错误后的重传成本和缓存需求，因此 MTU 是协议与工程条件的折中。

## 基本问题二：透明传输

透明传输表示任意比特组合都能作为数据传递，而不会被误认为帧边界或控制信息。

若协议使用特殊字节标记开始和结束，而该字节也出现在载荷中，接收端就可能过早结束帧。解决方法是在发送端对特殊内容进行转义，接收端再还原。

### 字节填充

发送端在定界符或转义字符之前插入转义字符（escape）；接收端看到转义字符后，把下一字节当作普通数据，而不是控制符。

![[Pasted image 20260715232024.png]]

> [!example] 转义字符本身也要转义
> 若 `ESC` 是转义字节，则数据中的 `ESC` 也编码成 `ESC ESC` 或协议规定的等价序列。接收端按规则删除额外转义，从而恢复原始数据。

### 比特填充

面向比特的协议可在特定连续比特模式中插入额外比特，避免载荷出现标志序列。接收端扫描并删除这些填充位。具体的五个连续 1 后插 0 的例子见[[3.2 点对点协议 PPP#透明传输]]。

## 基本问题三：差错检测

链路中的噪声和失真可能让 0 变成 1 或让 1 变成 0。误码率（Bit Error Rate, BER）是在观察期间错误比特数占传输比特总数的比例。数据链路层常在每帧尾部加入帧检验序列（Frame Check Sequence, FCS），接收端据此判断帧是否可能损坏。

以太网和 PPP 常使用循环冗余检验，详见[[3.1.2 循环冗余检验]]。

## 检错不等于可靠传输

> [!warning] “无比特差错”与“可靠交付”不同
> CRC 可以让接收端以很高概率拒绝含比特错误的帧，但不能自动发现或恢复帧丢失、重复和失序。可靠传输还需要序号、确认、超时、重传和去重等机制。

例如发送 `#1 → #2 → #3`，以下都属于传输错误：

- 丢失：`#1 → #3`；
- 重复：`#1 → #2 → #2 → #3`；
- 失序：`#1 → #3 → #2`。

这三个例子中的每个已收到帧都可能通过 CRC，却仍未实现可靠交付。

链路层是否提供重传取决于链路与协议设计。有些高质量点对点或交换式有线链路只检错并丢弃，把恢复交给上层；某些无线链路则在链路层加入确认和重传，以减少昂贵的端到端恢复。

## 本节小结

- 链路是物理通路，数据链路是链路加协议实现，帧是链路层数据单元。
- 成帧建立边界，填充机制保证控制标记不限制载荷内容。
- MTU 限制帧的上层数据部分，而不是整个帧。
- 检测比特错误不等于可靠传输，丢失、重复和失序需要额外机制处理。
