# 软件工程局部规则

- 区分需求、分析、设计、实现、测试、部署、运维和项目管理产物；不要把流程阶段机械写成固定瀑布顺序。
- 需求记录用户、目标、范围、约束和验收标准；不确定需求明确标记，不擅自替用户补充业务规则。
- 设计与架构笔记记录问题背景、质量属性、候选方案、取舍、决定和影响；设计模式必须说明适用条件与代价。
- 项目笔记保留实际技术栈、版本、目录、构建与运行方式；通用知识提炼到独立笔记后建立链接。
- 测试笔记说明测试层级、输入、预期结果、环境和覆盖边界；不得把未运行的测试写为“通过”。
- 故障记录区分现象、复现条件、证据、根因、修复和验证；没有证据时使用“可能原因”或“待核实”。
- 工具命令说明操作系统、Shell、版本和工作目录；涉及部署、删除、数据库迁移或远程系统时先确认影响范围。
- 课堂项目和个人实践保留原始要求、个人决策与复盘，不用通用模板大幅重写已有记录。
