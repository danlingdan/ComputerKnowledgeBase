# 14-计算机科学理论基础

> [!summary] 本章主线
> **问题 → 抽象模型 → 数据表示 → 算法 → 程序 → 机器执行**。硬件会更新，语言会流行或衰退，但抽象、算法与复杂度是理解计算机科学的稳定骨架。

## 1. 抽象与分层

**抽象（abstraction）**隐藏暂时不重要的细节，只暴露解决问题所需的接口。例如，调用 `sort(list)` 时不必先理解 CPU 如何移动每个字节。

| 层次 | 关注对象 | 示例 |
|---|---|---|
| 问题域 | 现实需求与约束 | 找出最短配送路线 |
| 算法 | 有限、明确、可执行的步骤 | Dijkstra 算法 |
| 程序 | 算法在语言中的实现 | Python 函数 |
| 运行时与操作系统 | 进程、内存、文件、系统调用 | 创建线程、读取文件 |
| 指令集 | 软件与处理器的接口 | load、add、jump |
| 数字逻辑 | 门电路与状态元件 | 加法器、触发器 |
| 物理器件 | 电信号与晶体管 | CMOS 晶体管 |

## 2. 信息、数据与编码

- **信息**是能够减少不确定性的内容；**数据**是信息的一种可记录表示。
- 一个二进制位（bit）有 0、1 两种状态；$n$ 位最多表示 $2^n$ 种状态。
- 整数常用二进制与二进制补码表示；实数通常采用浮点数近似表示。
- 文本采用字符编码，如 Unicode；图像可表示为像素矩阵；声音可通过采样和量化数字化。

> [!warning] 易错点
> 浮点数不是实数的精确副本。例如许多语言中 `0.1 + 0.2` 不能精确等于十进制 `0.3`。涉及金额时应使用定点数或十进制定点类型。

### 压缩、冗余与纠错

- **无损压缩**可完全恢复原数据，适合文本和程序；**有损压缩**允许舍弃感知不重要的信息，常用于图像、音频和视频。
- 冗余并非总是浪费：奇偶校验、校验和与纠错码用额外位发现或修复传输错误。

## 3. 逻辑与布尔代数

命题逻辑用真假值描述推理。基本运算包括 NOT（非）、AND（与）、OR（或）；它们既可表达程序条件，也可构造数字电路。

| A | B | A AND B | A OR B | NOT A |
|---:|---:|---:|---:|---:|
| 0 | 0 | 0 | 0 | 1 |
| 0 | 1 | 0 | 1 | 1 |
| 1 | 0 | 0 | 1 | 0 |
| 1 | 1 | 1 | 1 | 0 |

**德摩根律**：`NOT(A AND B) = (NOT A) OR (NOT B)`；它常用于化简条件表达式和电路。

## 4. 算法及其正确性

**算法（algorithm）**是解决一类问题的、有限且无歧义的步骤序列。一个合格算法应讨论：

1. 输入与输出是什么；
2. 是否对所有合法输入终止；
3. 输出是否满足规格；
4. 时间与空间成本是否可接受。

### 案例：线性查找与二分查找

在 $n$ 个元素中查找目标：

- 线性查找从头比较，最坏需要 $n$ 次，时间复杂度为 $O(n)$；无需预排序。
- 二分查找每次排除一半，最坏约需 $\log_2 n$ 次，时间复杂度为 $O(\log n)$；前提是数据有序且支持高效随机访问。

当 $n=1,000,000$ 时，线性查找最坏约比较一百万次，二分查找约 20 次。算法选择常比更换更快硬件更重要。

## 5. 数据结构

**数据结构（data structure）**组织数据并规定可执行的操作。选择依据不是“哪个最好”，而是工作负载需要什么操作。

| 结构 | 典型特征 | 常见用途 |
|---|---|---|
| 数组 | 连续、按下标访问快 | 表格、图像、向量 |
| 链表 | 节点通过引用连接 | 频繁插入删除 |
| 栈 | 后进先出（LIFO） | 函数调用、撤销 |
| 队列 | 先进先出（FIFO） | 任务调度、消息缓冲 |
| 哈希表 | 按键近似常数时间访问 | 字典、缓存、索引 |
| 树 | 层次结构 | 文件系统、语法树 |
| 图 | 顶点与边 | 路由、社交网络、依赖关系 |

## 6. 复杂度与规模效应

大 O 记号描述输入规模增长时资源消耗的数量级，忽略常数和低阶项。

| 复杂度 | 规模增大时的表现 | 典型操作 |
|---|---|---|
| $O(1)$ | 基本不随规模变化 | 数组下标访问 |
| $O(\log n)$ | 增长很慢 | 二分查找 |
| $O(n)$ | 线性增长 | 遍历列表 |
| $O(n\log n)$ | 通常可扩展 | 高效比较排序 |
| $O(n^2)$ | 大规模时迅速变慢 | 两两比较 |
| $O(2^n)$ | 仅适合较小输入 | 某些穷举搜索 |

> [!note] 分析边界
> 渐近复杂度不等于实际运行时间。输入分布、常数因子、缓存、并行度和实现质量同样重要，但复杂度能先判断方案是否可能扩展。

> [!info] 复杂度的深入入口
> 参见[[1.04.03 算法的时间复杂度|时间复杂度]]与[[1.04.04 算法的空间复杂度|空间复杂度]]；前者进一步区分问题规模、语句频度以及最好、最坏和平均情况，后者关注算法执行所需的辅助空间。

## 7. 可计算性与计算模型

**图灵机**是研究“什么可以通过机械步骤计算”的抽象模型。**Church–Turing 论题**认为，任何直觉上可有效计算的问题都能由图灵机计算；它是论题而非可由形式系统内部证明的普通定理。

有些问题不可判定。经典的**停机问题**表明：不存在一个通用算法，能对任意程序和输入总是正确判断该程序是否会停止。

复杂性理论进一步研究“能算”与“高效算”的差别。P 与 NP 问题询问：可快速验证答案的问题，是否也都能快速求解；截至本笔记更新时间它仍是公开问题。

## 8. 三种程序设计范式

- **命令式**：描述状态如何一步步改变，如 C、Python 的常规写法。
- **函数式**：强调函数组合、不可变数据与表达式求值，如 Haskell；许多现代语言吸收了其思想。
- **逻辑式/声明式**：描述目标或约束，让系统寻找解，如 Prolog、SQL 的查询表达。

范式不是互斥标签。工程中常在同一系统内组合使用。
