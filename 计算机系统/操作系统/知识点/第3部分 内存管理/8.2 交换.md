# 8.2 交换

本节聚焦于**交换**，是[[第八章 内存管理策略]]中的独立知识节点。

进程必须在内存中执行。进程可以暂时在从内存中交换(swap)到备份存储(backing store)。

## 8.2.1 标准交换

> [!info] 标准交换的定义与基本流程
> 
> **核心概念**：在物理内存与磁盘备份存储之间移动整个进程，以解决内存不足的问题。备份存储需要足够大，能容纳所有用户的内存映像。
> 
> **执行步骤**：当分派器发现下一个待运行进程不在内存时，如果没有空闲区域，则**换出（Swap out）**当前进程，**换入（Swap in）**所需进程，然后重新加载寄存器上下文并转移控制权。
        
2. **性能瓶颈与优化策略**
    
    - **高交换开销**：上下文切换时间主要取决于**数据传输时间**，且与内存大小成正比。公式示例：$100\text{MB}$ 进程在 $50\text{MB/s}$ 磁盘下，单向交换耗时 $2\text{s}$，总换入换出需要约 $4\text{s}$（$4000\text{ms}$）。
        
    - **按需优化**：应只交换进程**实际使用**的内存空间（而不是整个进程大小）。这要求进程使用 `request_memory()` 和 `release_memory()` 系统调用向操作系统动态报告内存需求变化。
        
3. **I/O 约束条件与双缓冲问题**
    
    - **约束条件**：如果进程正在等待异步 I/O 操作，绝对不能轻易换出，否则换入新进程 P_2 后，原 I/O 操作可能会错误地写入 P_2 的内存空间。
        
    - **解决方案**：① 不允许换出正在等待 I/O 的进程；② 采用**双缓冲（Double Buffering）**技术（I/O 先将数据写入操作系统内核缓冲区，待进程换入后再复制到用户内存中），但这会带来额外的数据复制开销。
        
4. **现代操作系统的实际应用**
    
    - **摒弃全量交换**：由于时间开销过高，现代操作系统不再直接使用原始的标准交换方案。
        
    - **变种实现**：在 UNIX、Linux 和 Windows 等现代系统中，通常采用交换的变种，并常与**虚拟内存**技术相结合（详见第9章）。
        
        - 变种一：**按阈值启用**（空闲内存低于阈值时启用，高于阈值时禁用）。
            
        - 变种二：**部分交换**（只交换进程的一部分而非整个进程，以降低交换时间）。
            

## 8.2.2 移动系统的交换

1. **移动系统禁用交换的原因**
    
    - 移动设备使用 **闪存** 而非传统机械硬盘作为永久存储。由于空间受限、闪存具有**写入次数寿命限制**，以及内存与闪存之间的**吞吐量差距较大**，移动操作系统设计者通常完全避免使用标准交换机制。
        
2. **iOS 的内存管理策略**
    
    - **拒绝交换**：从不将内存数据换出到闪存。
        
    - **内存不足时的处理**：请求应用程序**自愿释放**分配的内存。
        
    - **数据清理策略**：**只读数据**（如代码）允许被删除并在需要时从闪存重载；**已修改的数据**（如堆栈）不会被删除。
        
    - **终极手段**：对于无法释放足够内存的应用，操作系统将**直接终止**其运行。
        
3. **Android 的内存管理策略**
    
    - **拒绝交换**：同样避免使用交换机制。
        
    - **内存不足时的处理**：直接**终止进程**。
        
    - **状态保存与快速重启**：为了缓解杀进程带来的不良体验，Android 在终止进程前，会将该进程的**应用程序状态（application state）**写入闪存，使得再次启动时能够实现**快速恢复**。
        
4. **对开发者的约束与底层支持**
    
    - **开发者责任加重**：由于系统没有“交换”来兜底内存压力，移动开发者必须**极度小心**地分配和释放内存，避免内存泄漏或过度占用。
        
    - **底层依然具备能力**：尽管屏蔽了交换，但 iOS 和 Android 实际上**支持分页（paging）**技术，具备内存管理能力（会在后续章节详细探讨分页机制）。
