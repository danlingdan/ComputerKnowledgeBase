# 9.4 页面置换

本节聚焦于**页面置换**，是[[第九章 虚拟内存管理]]中的独立知识节点。

1. **过度分配（Over-Allocation）的风险**
    
    - 请求调页技术允许物理内存中同时容纳更多的进程，提高了多道程序设计的程度。
        
    - **场景**：例如，物理内存有 40 个帧，如果每个进程实际只需 5 个帧，我们可以运行 8 个进程（而不是按照满额 10 帧计算的 4 个进程）。
        
    - **风险**：一旦程序同时使用其实际所需的所有页面，就会出现**总需求内存（如 60 帧） > 物理可用内存（如 40 帧）**的情况，即发生过度分配。
        
2. **内存竞争与 I/O 缓存**
    
    - 物理内存不仅用于存放程序页面，大量的内存还被用于 I/O 缓存（缓冲区）。
        
    - 系统需要在**用户进程**和 **I/O 子系统**之间合理分配内存资源。分配失当会加剧内存压力，导致频繁的页面置换。
        
3. **缺页错误发生时若内存已满的困境**
    
    - 当进程执行时发生缺页错误，操作系统定位到所需页面在磁盘上的位置。
        
    - 如果此时空闲帧列表为空（**所有物理内存已被占用**），操作系统必须采取行动。
        
4. **操作系统的应对策略**
    
    - **终止用户进程**：这是一种极端且不推荐的方案，因为破坏了请求调页的**透明性**，用户会感到进程莫名崩溃。
        
    - **交换出整个进程**：将一个进程的所有帧换出到磁盘，释放大量物理内存。虽然可行，但在某些场景下代价高昂。
        
    - **页面置换（Page Replacement）**：这是_最常见且核心_的解决方案。即从当前内存中选出一个现有的页面（牺牲页）将其淘汰（换出到磁盘），空出帧位给新的缺页使用。后续章节将围绕此展开算法讨论。
        

## 9.4.1 基本页面置换

1. **基本页面置换流程**
    
    - 发生缺页错误时：
        
        1. 找到所需页面在磁盘上的位置。
            
        2. 若存在空闲帧，直接分配；若无，则通过置换算法选择**牺牲帧**（victim frame），将其内容写出（如果需要），并更新页表。
            
        3. 将所需页面读入空闲帧，更新页表。
            
        4. 重新启动被中断的进程。
            
    - **关键代价**：无空闲帧时，一次缺页错误需进行两次磁盘 I/O（换出牺牲页面 + 调入所需页面），使缺页处理时间加倍。
        
    
    > [!important] 基本页面置换流程
> 
> ```mermaid

>     A[Page Fault] --> B[Locate Page on Disk]
>     B --> C{Free Frame?}
>     C -->|Yes| D[Allocate Frame]
>     C -->|No| E[Select Victim]
>     E --> F{Dirty?}
>     F -->|Yes| G[Write to Disk]
>     F -->|No| H[Discard]
>     G --> D
>     H --> D
>     D --> I[Read New Page]
>     I --> J[Update Page Table]
>     J --> K[Restart Process]
>     
>     style C fill:#4CAF50,color:white
>     style F fill:#FF9800,color:white
> ```
> 
> **关键代价**：无空闲帧时，一次缺页错误需进行两次磁盘 I/O（换出牺牲页面 + 调入所需页面），使缺页处理时间加倍。

2. **修改位（脏位）优化**
    
    - 硬件维护一个**修改位（脏位）**，用于标记页面是否被修改过。
        
    - **优化机制**：当被选中作为牺牲帧时：
        
        - **被修改过（脏页）**：需要写回磁盘。
            
        - **未被修改（干净页/只读页）**：无需写回磁盘（磁盘已有副本），直接丢弃即可。
            
    - 这种机制可以将 I/O 时间减半，大幅降低缺页错误处理的开销。
        
3. **实现虚拟内存的两大核心算法**
    
    - **帧分配算法**：决定在多进程环境下，为每个进程分配多少个物理帧。
        
    - **页面置换算法**：在内存不足时，决定选择哪一个驻留页面作为牺牲帧。设计好算法至关重要，因为磁盘 I/O 开销极大。
        
4. **性能评估方法：引用串（Reference String）**
    
    - **方法**：通过记录程序的**内存地址访问序列**，并剥离页内偏移，提取出**页码序列**。
        
    - **简化规则**：
        
        1. 只考虑页号，忽略页内偏移量。
            
        2. 紧跟在 p 页引用后面的后续 p 页引用不会导致缺页错误，计算时需简化。
            
    - **示例**：地址序列 0100, 0432, ...，若页面大小为 100 字节，则可化简为引用串 1, 4, 1, 6, 1, 6, ...。
        
5. **缺页错误数量与帧数的关系**
    
    > [!note] Page Faults vs Frame Count
> 
> ```mermaid
> graph LR
>     subgraph FrameCount
>         direction LR
>         F1[1] --> F2[2] --> F3[3] --> F4[4] --> F5[5] --> Fn[n]
>     end
>     subgraph PageFaults
>         direction LR
>         P1[High] --> P2 --> P3 --> P4 --> P5 --> Pn[Low]
>     end
>     F1 --> P1
>     Fn --> Pn
>     style F1 fill:#f44336,color:white
>     style Fn fill:#4CAF50,color:white
> ```
    > 
    > - **图形化分析**：
    >   - 物理内存（帧数）越充足，缺页错误次数越少。
    >   - 在极端情况下（如只有 1 个帧时），每个内存引用都可能导致置换，缺页错误次数极多。
    >   - 典型关系呈现为一条下降的曲线（注：FIFO 算法可能出现 Belady 异常，即帧数增加时缺页率反而上升，但这不改变总体趋势）。
            
    - **示例引用串**：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1，将在后续章节用于测试不同的页面置换算法。
        

## 9.4.2 FIFO页面置换

1. **FIFO 算法的基本机制**
    
    - 基于**时间顺序**：记录每个页面调入内存的时间，需要置换时淘汰**最早调入**的页面。
        
    - 实现简单：利用一个 **FIFO 队列**，队首为最老页面，队尾为新调入页面。
        
2. **FIFO 算法的性能缺点与极端情况**
    
    - 容易选择错误的页面进行置换。
        
    - 可能淘汰掉**很久以前调入但已不再使用**的初始化模块（淘汰正确，但性能无提升）。
        
    - 也可能淘汰掉**包含重要变量且仍在频繁使用**的活动页面（淘汰错误），导致刚被淘汰又被立即调用，引发频繁的缺页错误，性能大幅下降。
        
    - **重要提示**：虽然会导致性能低下，但该算法**绝不会导致执行错误**，仅仅是性能问题。
        
3. **Belady 异常（Belady's Anomaly）**
    
    - **定义**：对于某些页面置换算法（如 FIFO），当分配给进程的**物理帧数量增加时，缺页错误次数反而增多**。
        
    - **颠覆直觉**：原本普遍认为，给进程分配越多的物理内存，缺页错误越少。Belady 异常证明这种直觉并非对所有算法都成立（通常栈式算法不会出现此现象，而 FIFO 是非栈式算法）。
        
    - **具体案例**：针对引用串 1,2,3,4,1,2,5,1,2,3,4,5，分配 3 个帧时产生 9 次缺页错误；分配 4 个帧时反而产生了 10 次缺页错误。这也是早期虚拟内存研究中的重大发现。
        
        > [!warning] Belady 异常示例
> 
> **引用串**：1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5
> 
> | 帧数 | 缺页错误次数 |
> |------|-------------|
> | 3 | 9 |
> | 4 | 10 |
> 
> ```mermaid
> graph TD
>     A[3 Frames: 9 Faults] --> B[4 Frames: 10 Faults]
>     B --> C[Frames ↑, Faults ↑]
>     style B fill:#f44336,color:white
> ```

## 9.4.3 最优页面置换

1. **OPT / MIN 算法的核心规则**
    
    - 它提出了一个理想化的置换决策：**淘汰在未来最长时间内不会被访问到的页面**。
        
    - 该算法不会发生 Belady 异常，并能提供理论上的最低缺页错误率。
        
2. **性能优势示例**
    
    - 以引用串 7,0,1,2,0,3,0,4,2,3,0,3,2,1,2,0,1,7,0,1 为例，通过 OPT 算法，在仅使用 3 个物理帧的情况下，仅产生 **9 次缺页错误**。
        
        > [!example] OPT 算法性能对比
> 
> **引用串**：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1
> 
> | 算法 | 3帧缺页次数 |
> |------|-----------|
> | OPT | 9 |
> | LRU | 12 |
> | FIFO | 15 |
> 
> ```mermaid
> graph TD
>     A[OPT: 9] --> B[LRU: 12] --> C[FIFO: 15]
>     style A fill:#4CAF50,color:white
>     style C fill:#f44336,color:white
> ```
> 
> 相比之下，FIFO 算法在该例子中产生了 **15 次缺页错误**。最优置换算法的性能差不多是 FIFO 算法性能的一倍。
        
    - 该引用串在 3 个帧下的理论缺页错误下限就是 9 次。
        
3. **算法的局限性与实际用途**
    
    - **无法实现**：OPT 算法需要预知程序未来的所有内存访问序列，这在现实的计算机系统中是==不可能==的。
        
    - **衡量标准（标杆）**：它主要用于学术研究和性能评估。通过将实际算法（如 LRU、LFU、FIFO 等）与 OPT 算法进行对比，计算两者之间的偏差百分比（如“最坏不劣于 12.3%”或“平均不劣于 4.7%”），可以客观评估实际算法的优劣。
        

## 9.4.4 LRU页面置换

1. **设计思想（OPT 的实用近似）**
    
    - **LRU (最近最少使用)** 基于“过去的最近使用情况可以近似未来的访问情况”的假设，置换**最长时间没有使用**的页面。
        
    - 它本质上是 OPT 算法在**时间上向后看**（分析历史）的替代方案，而不是向前看（预测未来）。对于倒转引用串 S^R，OPT 和 LRU 的缺页错误率保持不变。
        
2. **性能表现**
    
    > [!info] LRU Performance
> 
> ```mermaid
> graph TD
>     subgraph LRU_Stack
>         S1[Most Recent]
>         S2[]
>         S3[]
>         S4[LRU]
>     end
>     S1 --> S2 --> S3 --> S4
>     style S1 fill:#4CAF50,color:white
>     style S4 fill:#f44336,color:white
> ```
    > 
    > - 在示例引用串中，LRU 产生了 **12 次**缺页错误。
        
    - 性能明显优于 FIFO (15 次)，但不如理想的最优算法 OPT (9 次)。
        
3. **关键特性：堆栈算法与无 Belady 异常**
    
    - LRU 属于**堆栈算法**（Stack Algorithm）。
        
    - 这意味着它**绝不会发生 Belady 异常**（即增加分配给进程的物理帧数量，不会导致缺页错误率上升）。因为对于 LRU，n 个帧的内存集合是 n+1 个帧内存集合的子集，增加帧只会追加最常用的页面。
        
4. **实现的巨大挑战（核心难点）**
    
    - 由于需要精确记录页面的“最近使用”顺序，LRU 的实现**极度依赖硬件辅助**（除了标准的 TLB），纯软件实现会带来不可接受的性能损耗。
        
    - **方法一：计数器（时间戳）**
        
        - 为 CPU 添加全局逻辑时钟，每次内存访问递增，并将当前时间写入页表条目。
            
        - 置换时扫描页表寻找最小时间值。
            
        - **缺点**：每次内存访问都要写内存，扫描页表耗时，且需处理时钟溢出。
            
    - **方法二：堆栈（双向链表）**
        
        - 维护一个双向链表栈。最近使用的页面在栈顶，LRU 在栈底。每次访问需将页面从中间移除并放到栈顶（最差需修改 **6 个指针**）。
            
        - **缺点**：虽然置换时无需搜索，但每次内存引用仍要执行链表移除/插入操作。
            
5. **性能警告**
    
    - 必须使用硬件支持（如 TLB 中的引用位等近似策略）。如果试图通过**中断**（Trap）进入操作系统，由软件来更新这些数据结构，会导致内存访问速度降低 **至少 10 倍**，这在现代计算机系统中是完全不可容忍的。
        

## 9.4.5 近似LRU页面置换

1. **真实 LRU 的实现困境**
    
    - 现实中，极少有系统能够提供充足的硬件支持来实现**真正的 LRU（最近最少使用）**算法（如计数器方案或堆栈方案）。
        
    - 硬件能力不足时，只能退而求其次，使用 FIFO 或其他基础算法。
        
2. **引用位（Reference Bit）硬件支持**
    
    - 多数系统通过引入 1 位的**引用位**来提供轻量级支持。
        
    - **操作规则**：
        
        - 操作系统会周期性地将所有页面的引用位**清零**（设为 0）。
            
        - 硬件在**每次**页面被访问（读或写）时，会自动将对应页面的引用位**置为 1**。
            
3. **近似 LRU 的基础**
    
    - 系统可以通过检查引用位来判断页面“近期是否被使用过”，但无法得知“被使用的先后顺序”。
        
    - 这种有限的“使用/未使用”信息，并不是真正的 LRU，但它是构建“近似 LRU”算法（如**二次机会算法**、**增强型二次机会算法**等）的基石，在实际操作系统中极为常见。
        

### 9.4.5.1 额外引用位算法

1. **硬件辅助机制**
    
    - 为内存中每个页面额外维护一个 **8 位的移位寄存器**，用于记录过去 **8 个周期**内的页面使用情况。
        
    - 由定时器（比如每 $100\text{ms}$）触发操作系统动作，将当前的硬件引用位移入寄存器**高位**，所有旧位右移，最低位舍弃。
        
2. **评分与置换策略**
    
    - 寄存器值反映了历史访问密度：`00000000` 表示最近从未使用；`11111111` 表示每个周期都被使用。
        
    - 比较标准：将该 8 位二进制数视为**无符号整数**。
        
    - 置换规则：**数值越小**的页面，代表“最近最少使用”（LRU），优先被置换。
        
    - 平局处理：若多个页面拥有相同的最小值，则在这些页面之间采用 **FIFO** 算法作为辅助策略。
        
3. **自适应与极限简化**
    
    - 历史记录长度（即寄存器位数）可根据系统硬件能力灵活调整，旨在兼顾更新速度与精度。
        
    - **极限情况**：当位数缩减为 **0 位** 时，该算法仅基于当前的单一引用位进行判断，这实际上退化为著名的 **第二次机会页面置换算法**（Second-Chance algorithm）。
        

### 9.4.5.2 第二次算法机会

1. **算法核心思想**
    
    - 它是 FIFO 算法的**改进版本**。通过引入**引用位**（Reference Bit），给原本需要被淘汰的页面提供“第二次机会”，避免刚调入内存的活跃页被立即置换。
        
    - 经常使用的页面（引用位一直为 1）不会被淘汰。
        
2. **执行逻辑**
    
    - 检查被选中作为牺牲者的页面引用位：
        
        - 若为 0，直接淘汰该页。
            
        - 若为 1，则清除其引用位（置为 0），将其到达时间更新为当前时间，并视为获得“第二次机会”，继续检查队列中的下一个 FIFO 页面。该页面会被移到队列尾部，免受本次淘汰。
            
3. **时钟算法（Clock Algorithm）的具体实现**
    
    - 采用**循环队列**，并维护一个**时钟指针**。
        
    - 置换发生时，指针向前（顺时针）移动，寻找引用位为 0 的页面。
        
    - 在指针移动过程中，遇到引用位为 1 的页面时，会将引用位清零（这就相当于给它们二次机会），指针继续前进。
        
4. **最坏情况与退化**
    
    - 假如循环队列中所有页面的引用位都是 1，时钟指针将绕队列**旋转一整圈**，将每个页面的引用位清零。
        
    - 如果转完一圈依然没有找到引用位为 0 的页面（即回到原点时所有位都被清为 0了），算法便会**退化为标准的 FIFO 置换**，淘汰当前指针指向的页面。
        

> [!important] 时钟算法（Clock Algorithm）
> 
> ```mermaid
> graph LR
>     subgraph Clock_Queue
>         P1[P1: R=1] --> P2[P2: R=0] --> P3[P3: R=1] --> P4[P4: R=1]
>         P4 --> P1
>     end
>     P2 --> X[选择P2淘汰]
>     style P2 fill:#f44336,color:white
> ```
> 
> **工作原理**：循环队列 + 时钟指针。指针移动时，遇到 R=1 的页面则清为 0（给第二次机会），继续前进；找到第一个 R=0 的页面淘汰。

### 9.4.5.3 增强型第二次机会算法

1. **核心机制与位组合** 增强型第二次机会算法并不是仅仅依赖单一的“引用位”，而是将**引用位**和**修改位（脏位）**组合成一个有序对 (参考位, 修改位)，以此将页面分为四个等级（优先级由高到低，即置换优先级由高到低）：
    
    - **(0, 0)**：最近未使用，且未修改。**最佳置换目标**。置换时无需写回磁盘（节约 I/O）。
        
    - **(0, 1)**：最近未使用，但已修改。**次要置换目标**。置换前需将脏数据写回磁盘。
        
    - **(1, 0)**：最近使用过，但未修改。不建议立刻置换，因为可能很快再次使用。
        
    - **(1, 1)**：最近使用过，且已修改。**最不建议置换**，因为不仅可能很快再次使用，置换前还需执行磁盘写入 I/O。
        
2. **算法执行逻辑**
    
    - 采用与**时钟算法（Clock Algorithm）**类似的循环队列结构。
        
    - 淘汰策略：寻找**非空的最低类型**（即优先寻找 (0,0) 页面）的第一个页面进行淘汰。
        
    - 寻优过程：为了找到合适的置换页面，可能需要**多次扫描**循环队列。在多次扫描过程中，算法通常会将页面的引用位清零，从而提升其在后续扫描中被淘汰的概率。
        
3. **主要优势：减少 I/O 开销**
    
    - 与简单的时钟算法相比，增强了**对“脏页”（已修改页面）的保护**。
        
    - 优先置换干净的未修改页面 (0,0)，最大程度地避免了不必要的磁盘写入操作，从而显著**降低了系统所需的磁盘 I/O 数量**，提升了虚拟内存的整体性能。
        

## 9.4.6 基于计数的页面置换

1. **基于计数的置换算法原理**
    
    - 系统为每个物理内存中的页面维护一个访问计数器，记录该页面被引用的次数。
        
2. **LFU（最不经常使用）算法**
    
    - **规则**：置换引用计数**最小**的页面。
        
    - **缺点**：存在“**历史遗留问题**”。例如，一个页面在程序启动时被大量使用（导致计数很高），但之后程序逻辑改变，该页面不再需要，却因为高计数而长期驻留在内存中，无法被置换。
        
    - **改进方案**：定期将计数器右移 1 位（相当于除以 2），实现**指数衰减**，使早期的引用权重逐渐降低，避免历史遗留问题。
        
3. **MFU（最经常使用）算法**
    
    - **规则**：置换引用计数**最大**的页面。
        
    - **论点**：计数最小（甚至为 0）的页面极有可能是刚刚被调入内存且还没使用过的页面，它们基于“局部性原理”在将来很可能马上要被用到。
        
4. **算法评价与现实应用**
    
    - **局限性**：LFU 和 MFU 都需要为每个页面维护计数器，并且在置换时需要遍历页面查找最大/最小计数值，**实现开销昂贵**。
        
    - **预测能力差**：这两种算法在实际测试中，都**不能很好地近似最优（OPT）置换算法**，因此在实际操作系统中很少被采用。
        

## 9.4.7 页面缓冲算法

1. **核心思想与动机**
    
    - 页面缓冲算法并不是一个独立的替代算法，而是对**任意页面置换算法**的一种通用增强措施。
        
    - 其核心目标是：通过引入**空闲帧缓冲池**及相关辅助机制，抵消置换策略错误带来的额外 I/O 开销，提高性能。
        
2. **三大核心增强机制**
    
    **① 快速重启机制（预先分配）**：
    
    - 发生缺页时，进程直接使用空闲缓冲池中的帧，**无需等待**旧牺牲帧的写入操作。这使得缺页处理时间大幅缩短，进程能迅速恢复运行，后台再逐步将牺牲帧写回磁盘并补充到空闲池。
        
    
    **② 脏页后台写回（减少置换时阻塞）**：
    
    - 系统维护一个**修改页面列表（脏页列表）**。当 I/O 设备闲置时，主动将脏页写入磁盘并清空“修改位”。这极大地增加了置换发生时找到“干净页面”的概率，从而减少了置换时触发磁盘 I/O 的次数。
        
    
    **③ 回收缓存机制（免 I/O 恢复）**：
    
    - 空闲缓冲池不仅保留帧，还**保留页面对应关系**。如果一个页面刚被置换出去（变为空闲帧）但又被进程立即访问，系统可直接从缓冲池中**原位取回**，完全避免了不必要的磁盘 I/O。
        
3. **实际应用场景**
    
    - **VAX/VMS 系统（搭配 FIFO 算法）**：早期 VAX 硬件缺乏有效的“引用位”硬件支持，只能使用简单的 FIFO。虽然 FIFO 容易错误地置换出热点常用页面，但缓冲池的**回收缓存机制**能立刻纠错（无 I/O 取回），从而弥补了基础算法的不足。
        
    - **UNIX 系统（搭配第二次机会算法）**：将该机制与传统算法结合，进一步提升系统的整体内存置换性能。
        
4. **整体优势**
    
    - 这种方法具备极强的**普适性**，可以附加在任何页面置换算法上，通过减少对物理磁盘的直接读写，大幅降低因置换算法误判而产生的性能惩罚。
        

## 9.4.8 应用程序与页面置换

1. **通用操作系统算法与特定应用程序的冲突**
    
    - 操作系统提供的通用内存管理与 I/O 缓冲，对于像**数据库**这样拥有深度定制内存及 I/O 管理策略的程序而言，不仅没有帮助，反而会造成资源浪费（如**双重缓冲**导致内存倍增）。
        
2. **页面置换算法在特定场景下的失效**
    
    - **数据仓库应用**通常执行**顺序读取**（大块读取旧数据）后计算写入。
        
    - **LRU 算法（最近最少使用）** 倾向于保留新数据并淘汰旧数据，这完全违背了数据仓库“优先读取旧数据”的访问模式，导致性能低下。
        
    - 在此类场景下，**MFU 算法（最经常使用）** 反而比 LRU 表现得更高效。
        
3. **绕过虚拟内存与文件系统：原始磁盘（Raw Disk）**
    
    - 为了解决上述矛盾，操作系统允许将磁盘分区直接作为**原始磁盘**使用，通过**原始 I/O** 进行访问，**完全绕过文件系统**（包括请求调页、文件锁定、预取、空间分配、目录结构等）。
        
    - 这种方法相当于让应用程序直接和磁盘硬件打交道，可以获得极高的 I/O 性能。
        
    - **实际权衡**：虽然少数需要极高性能的专用应用（如某些大型数据库）适合使用原始 I/O，但对于绝大多数普通应用程序，使用操作系统提供的**通用文件系统服务**仍然是更优、更安全的选择。
