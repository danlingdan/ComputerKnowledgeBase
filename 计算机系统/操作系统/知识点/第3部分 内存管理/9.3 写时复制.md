# 9.3 写时复制

本节聚焦于**写时复制**，是[[第九章 虚拟内存管理]]中的独立知识节点。

1. **引入写时复制（CoW）的动机**
    
    - 传统的 `fork()` 系统调用会为子进程**完整复制**父进程的整个地址空间，这非常耗时且浪费物理内存。实际使用中，子进程往往在创建后立即调用 `exec()` 来执行新程序，此时完整的地址空间复制是毫无意义的。
        
    - **写时复制（CoW）** 技术旨在解决此问题：提供快速进程创建，并最小化新进程所需的物理页面分配。
        
2. **写时复制（CoW）的核心机制**
    
    - 父进程和子进程初始时**共享相同的物理页面**（而非复制页面）。
        
    - 这些共享页面会被标记为**写时复制**（Copy-on-Write）。只要两者都是只读，就共享同一物理内存。
        
    - **触发复制**：当父进程或子进程中的任何一方试图**写入**某个共享页面时，操作系统才会真正分配一个新的物理页面，复制旧页面的内容，并将写入操作重定向到副本上。
        
    - 对于**不可修改的页面**（如纯代码段），则可以被两个进程永久共享，完全不需要复制。
        
3. **页面分配策略**
    
    - 操作系统维护一个**空闲页面池**（page pool），专门用于 CoW 触发时的页面分配。
        
    - 分配出的新页面通常采用**按需填零（Zero-fill-on-demand）**技术：即在分配前先将内存清空为 0。这是为了防止新分配的页面残留之前进程的敏感数据，提高系统安全性。
        
4. **`vfork()` 变种**
    
    - 在部分 UNIX 系统（如 Solaris、Linux）中，引入了更激进的 `vfork()` 系统调用。
        
    - **工作原理**：在创建子进程时，父进程会被**挂起**（暂停执行），子进程直接借用父进程的整个地址空间，且**不进行**写时复制保护。
        
    - **风险与限制**：因为子进程修改的页面会直接影响到父进程，所以使用 `vfork()` 时必须非常小心，子进程中**严禁**修改父进程的任何页面。
        
    - **适用场景**：由于不复制任何页面，`vfork()` 效率最高，通常专门用于子进程在创建后**立即调用 `exec()`** 的场景（如 UNIX Shell 实现命令解析时）。
