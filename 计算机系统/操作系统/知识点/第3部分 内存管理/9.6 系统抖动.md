# 9.6 系统抖动

本节聚焦于**系统抖动**，是[[第九章 虚拟内存管理]]中的独立知识节点。

1. **抖动的定义与成因**
    
    - **抖动（Thrashing）**：指系统由于物理内存极度不足，导致进程频繁发生缺页错误，从而进行极高频率的页面换入换出的现象。
        
    - **根本原因**：分配给进程的物理帧数不足以容纳其当前**工作集**（当前正在活跃使用的页面集合）。
        
2. **抖动的恶性循环**
    
    - 进程缺页 -> 必须换出某个页面 -> 由于所有页面都在使用，刚被换出的页面立刻又被需要 -> 再次缺页错误 -> 再次换出，**频繁在内存与磁盘之间进行 I/O 传输**，陷入死循环。
        
3. **抖动状态的判断标准**
    
    - 最直接的判断标准是：**进程用于调页（交换）的时间超过了它实际用于执行指令（计算）的时间**。一旦发生这种情况，系统的有效吞吐量会急剧下降，甚至接近零。
        
4. **应对与缓解手段**
    
    - 文中提出了一种通过**中级 CPU 调度（换进换出层）** 进行干预的方法。
        
    - 具体策略：对于低优先级的进程，如果发现其分配的帧数低于结构要求的最低数量，就将它**挂起（暂停执行）**，并将其所有的剩余页面**调出**到磁盘，从而释放出宝贵的物理内存帧，分配给其他正在运行的进程，以缓解整体的内存压力。
        

## 9.6.1 系统抖动的原因

1. **抖动的诱因与恶性循环**
    
    - **定义**：当物理内存严重不足，导致进程频繁发生缺页错误，CPU 绝大部分时间用于处理磁盘 I/O（调页），几乎无法执行有效计算的严重性能瓶颈现象。
        
    - **触发机制**：操作系统为了提高 CPU 利用率而增加**多道程序程度**（引入更多进程）。但若使用**全局置换算法**，一个进程的缺页会抢夺其他进程的物理帧，导致连锁反应，大量进程同时陷入频繁缺页的泥潭。
        
    - **恶性循环**：CPU 利用率下降 → 调度程序误判以为进程不够多，继续增加多道程序程度 → 系统更严重地抖动 → CPU 利用率进一步骤降，系统吞吐量趋向于 0。
        
2. **应对策略与解决方案**
    
    - **核心对策**：一旦检测到**多道程序程度过高引起的抖动**，必须采取行动**降低多道程度**（例如挂起或终止部分进程，释放其内存）。
        
    - **算法约束**：采用**局部置换算法**可以限制抖动扩散，避免一个抖动进程抢夺其他进程的帧。但它不能完全解决排队 I/O 等待时间过长导致的整体性能下降问题。
        
3. **局部性模型（Locality Model）及其作用**
    
    - **局部性定义**：进程对内存的访问不是随机的，而是呈现出明显的聚集性（局部）。程序在某一时间段内频繁访问的内存页范围被称为一个**局部**。当函数调用、循环发生时，局部会随之改变。
        
    - **理论基石**：局部性是**缓存**和虚拟内存**工作集**策略有效运行的根本原理。如果是随机访问，缓存机制就失去了意义。
        
    - **防抖动的核心原则**：为了防止抖动，系统必须为进程分配**足够的物理帧以容纳当前执行阶段的活动局部**。当局部发生切换时，允许发生短暂的缺页；但一旦局部内的所有页面全部装入内存，除非再切换局部，否则不应该再出现缺页。如果分配的帧数连当前局部都装不下，就会立刻引起抖动。

## 9.6.2 工作集模型

1. **工作集模型的定义**
    
    - 基于**程序局部性原理**，引入参数 $\Delta$ 定义**工作集窗口**（观察过去 $\Delta$ 个内存引用）。
        
    - 在这段时间内被访问过的页面集合即为当前进程的**工作集（Working-Set）**。
        
    - 未在 $\Delta$ 时间内被访问的页面会从工作集中移除（代表其已失活）。
        
2. **工作集的动态性与精度**
    
    > [!note] 工作集动态变化
    > 
    > ```mermaid
    > graph TD
    >     subgraph WS_t1
    >         WS1["WS(t₁) = {1,2,5,6,7}"]
    >     end
    >     subgraph WS_t2
    >         WS2["WS(t₂) = {3,4}"]
    >     end
    >     subgraph WS_t3
    >         WS3["WS(t₃) = {8,9}"]
    >     end
    >     WS_t1 --> WS_t2 --> WS_t3
    >     style WS1 fill:#4CAF50,color:white
    >     style WS2 fill:#FF9800,color:white
    >     style WS3 fill:#2196F3,color:white
    > ```
    > 
    > - 工作集是随着程序执行（进入不同的局部）而动态变化的，如 WS(t₁) = \{1,2,5,6,7\} 变到 WS(t₂) = \{3,4\}。
        
    - $\Delta$ 的选择决定了精度：太小无法容纳当前局部；太大会包含不连续的多段局部；若 $\Delta$ 无限大，则工作集等于整个程序段。
        
3. **工作集与防止抖动的机制**
    
    - 设进程 i 的工作集大小为 $WSS_i$，系统中所有进程的总帧需求为 $D = \sum WSS_i$。
        
    - **分配策略**：操作系统监控每个进程的工作集，并动态分配与其工作集大小相当的物理帧。
        
    - **检测系统抖动**：当 **D > m**（系统总帧数 m）时，意味着内存严重不足，系统必将进入**抖动**（Thrashing）状态。
        
    - **解决方案**：系统必须挂起（swap out）一个或多个进程，释放其占用的物理帧分配给其他进程，从而在防止抖动的条件下最大化系统的多道程序程度。
        
4. **实现难点与近似方案**
    
    - **难点**：跟踪动态窗口中的每次内存引用成本极高。
        
    - **硬件近似方案**：使用**定时器中断**和**引用位（Reference Bit）**来近似。
        
    - **具体实现**：例如取 $\Delta$ 为 10000 次引用，每 5000 次引用触发中断。系统记录并重置引用位。缺页发生时，通过检查当前引用位和保留的历史位，判断页面是否在过去的 10000 \sim 15000 次引用中被访问过。
        
    - **精度与成本的权衡**：增加历史位（如 10 位）和提高中断频率（如每 1000 次）可以极大提高工作集估算的准确性，但也会带来更高的 CPU 中断处理开销。
        

## 9.6.3 缺页错误频率

1. **PFF 策略的优势与核心思想**
    
    - 工作集模型虽然有效，但计算和实现略显笨拙。**缺页错误频率（PFF）** 策略提供了一种更直接、更易操作的防止系统抖动的方案。
        
    - 其核心逻辑是：**直接监控进程的缺页错误率**，因为系统抖动最直接的表现就是缺页错误率骤升。
        
2. **动态帧数调整（基于上下限）**
    
    > [!important] PFF Dynamic Frame Allocation
    > 
    > ```mermaid
> graph TD
>     A[Page Fault Rate] --> B{GT Upper?}
>     B -->|Yes| C[Allocate Extra Frame]
>     B -->|No| D{LT Lower?}
>     D -->|Yes| E[Remove One Frame]
>     D -->|No| F[Keep Unchanged]
>     style B fill:#f44336,color:white
>     style D fill:#4CAF50,color:white
> ```
    > 
    > **触发条件与处理**：
    > - 当缺页率 **超过上限** -> 说明给该进程分配的帧数太少，导致频繁缺页 -> 系统为进程**再分配一个帧**。
    > - 当缺页率 **低于下限** -> 说明给该进程分配的帧数远大于其实际需求（存在内存浪费） -> 系统从进程中**删除一个帧**，让出资源给其他进程。
            
3. **防止抖动的终极手段（换出进程）**
    
    - 如果系统检测到某个进程缺页率飙升需要额外帧，但此时**物理内存已经用完（没有空闲帧）**，系统必须采取强硬手段。
        
    - 具体做法是：**选择另一个进程挂起并换出到后备存储（磁盘交换区）**，将释放出的帧分配给当前缺页率过高的进程。这种策略与工作集模型中的挂起逻辑是一致的。
