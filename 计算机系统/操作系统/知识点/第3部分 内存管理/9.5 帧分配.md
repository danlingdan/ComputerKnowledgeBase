# 9.5 帧分配

本节聚焦于**帧分配**，是[[第九章 虚拟内存管理]]中的独立知识节点。

1. **帧分配的核心问题**
    
    - 在多进程环境中，操作系统面临的核心挑战是：如何将有限的物理内存（即物理帧）合理地分配给各个正在运行的进程。
        
    - 示例场景：现有 93 个空闲帧，需要分配给 2 个进程，如何决定各自的配额。
        
2. **单用户系统下的基础分配模型**
    
    - **物理内存配置**：系统总内存 $128\text{KB}$，页面大小为 $1\text{KB}$，总共有 128 个物理帧。操作系统自身占用 $35\text{KB}$（即 35 帧），剩余 **93 帧** 完全分配给用户进程使用。
        
    - **纯请求调页过程**：进程初始运行时，空闲帧列表包含全部 93 帧。进程发生的**前 93 次缺页错误**会直接从空闲列表获取帧。当 93 帧用满后，后续发生的缺页错误将**触发页面置换**（从已装入的 93 页中淘汰一个旧页，腾出空间给新页）。
        
    - **生命周期管理**：当用户进程终止时，其占用的 93 个帧会**被释放并归还**到空闲帧列表中，供下一个进程使用。
        
3. **分配策略的变种与优化保障**
    
    - **操作系统缓冲区复用**：操作系统不仅占据自己的固定内存，也可以预先从空闲帧池分配缓冲空间。在不被操作系统使用时，这些空间可以暂时出借给用户进程的调页操作使用。
        
    - **保留最小空闲帧池（关键保障）**：为了防止因空闲帧耗尽而导致页面置换必须同步进行（阻塞等待写出），系统通常会保留**至少 3 个空闲帧**作为缓冲池。这确保了在发生缺页错误时，进程可以立即获得一个空闲帧进行读取，而将旧页的写回操作放在后台异步完成。
        
    - **核心原则总结**：尽管有诸多变体，帧分配的基本策略非常清晰——**用户进程会动态分配到系统中可用的任何空闲帧，直至物理内存被完全占用。**
        

## 9.5.1 帧的最小数

1. **帧分配的下限与性能关联**
    
    - 在多进程系统中，给每个进程分配的物理帧数量不能低于某个**最小阈值**。
        
    - **性能衰减**：分配的帧越少，进程发生缺页错误的概率就越高，这会严重降低进程的执行速度。
        
2. **指令原子性与重启的要求（核心原因）**
    
    - 操作系统必须能够在执行一条机器指令的过程中处理缺页错误，并**完整地重新启动该指令**。
        
    - 为了支持指令的透明重启，操作系统必须为该条指令在**执行时可能触及的所有不同页面**分配足够的物理帧。
        
3. **最小帧数的硬件依赖性（由架构决定）**
    
    - 最小帧数并不是由操作系统主观设定的，而是由**CPU 的指令集架构（ISA）**所决定的硬件物理需求。
        
    - **简易寻址案例**：如果一个指令只需要获取指令本身和一个操作数，至少需要 2 个帧（1 个指令帧 + 1 个操作数帧）；一级间接寻址则至少需要 3 个帧。
        
    - **复杂指令案例**：
        
        - **PDP-11 移动指令**：指令跨 2 页，操作数为 2 个且都支持间接寻址，共需要 **6 个帧**。
            
        - **IBM 370 MVC 指令**：指令本身跨 2 页，源数据和目的数据各跨 2 页，基础情况需要 **6 个帧**；当该指令作为跨页 EXECUTE 指令的操作数时，最坏情况需要 **8 个帧**。
            
4. **多级间接寻址的极端情况与限制**
    
    - **理论最坏情况**：如果指令支持无限多级的间接寻址（即指针指向指针...），理论上一条简单的加载指令可能导致全虚拟内存的所有页面都被引用，这意味着整个虚拟内存都必须驻留在物理内存中，这显然是不可能的。
        
    - **硬件防陷阱机制**：现代体系结构会通过硬件来**限制间接寻址的级数**（例如限制最多 16 级）。每当发生间接引用时递减计数器，若减至 0 则触发“过度间接引用”陷阱。
        
    - **最大帧数界定**：这种限制保证了单条指令所需的最大帧数是有上限的（例如限制后为 17 个）。
        
5. **总结**
    
    - 分配的帧数必须保持在**由体系结构决定的最小帧数**和**由物理内存总量决定的最大帧数**之间。在这两个极端数值之间，操作系统拥有广泛的帧分配算法选择空间。
        

## 9.5.2 分配算法

1. **平均分配（Equal Allocation）**
    
    - **机制**：将 m 个可用物理帧平均分配给 n 个进程，每个进程获得 m/n 个帧。
        
    - **缺点**：无视进程实际内存需求（局部性差异），容易导致小进程占用多余帧造成浪费，而大进程帧数不足导致频繁缺页。
        
2. **比例分配（Proportional Allocation）**
    
    - **机制**：根据进程的虚拟内存大小 $s_i$ 进行按比例分配。总内存 $S = \sum s_i$，目标帧数为 $m$。进程 $p_i$ 分得帧数 $a_i$ 依据公式：
        
    $$
    a_i = \frac{s_i}{S} \times m
    $$
        
          
        
    - **优势**：克服了平均分配的缺点，使得大内存需求的进程获得更多物理帧，小内存需求的进程获得较少物理帧，内存利用更为合理（文中示例中，10 页进程得 4 帧，127 页进程得 57 帧）。
        
3. **动态调整机制**
    
    - 当系统中多道程序设计的程度发生变化时，各个进程分配的帧数也会动态调整：
        
        - **多道程度增加**：调入新进程，原有各进程让出部分帧给新进程。
            
        - **多道程度降低**：进程退出，其占用的帧重新分配给剩余进程。
            
4. **优先级问题的考虑**
    
    - 基础的平均分配和比例分配算法**一视同仁**地对待不同优先级的进程。
        
    - **改进方向**：在系统资源受限时，一种更优的策略是按照**进程的优先级、或者优先级与进程大小的组合**来分配物理内存，以确保高优先级任务能够获得更充足的内存，从而得到更快的执行速度。
        

## 9.5.3 全局分配与局部分配

1. **两种核心置换策略**
    
    - **全局置换（Global Replacement）**：进程缺页时，可以从系统中**所有物理帧**中选择牺牲帧（包括其他进程已占用的帧）。即进程可以“抢走”其他进程的帧。
        
    - **局部置换（Local Replacement）**：进程缺页时，只能从**分配给该进程自身的固定帧集合**中选择牺牲帧，不会影响其他进程的帧分配。
        
2. **优先级与帧抢夺**
    
    - 全局置换天然适合实现优先级调度。**高优先级进程**可以通过全局置换机制，从低优先级进程那里“夺取”物理帧，从而获得更快的内存访问速度。
        
    - 在局部置换下，无论进程优先级高低，分配给它的帧数量不会动态改变。
        
3. **全局置换的弊端（外部依赖性）**
    
    - **不可控的缺页率**：进程的缺页频率不仅取决于自身行为，还严重依赖于系统中其他进程的行为。
        
    - **执行时间差异巨大**：由于环境的变化，相同任务在不同时刻执行的时间可能相差极大（文本举例为 $0.5\text{s}$ vs $10.3\text{s}$），这给系统的实时性和性能预测带来了不确定性。
        
4. **局部置换的弊端（局限性）**
    
    - 即使系统中有大量其他进程不再使用的空闲内存，局部置换策略也会阻止某个缺页严重的进程去使用这些内存。这会造成物理内存的**利用不充分**，导致进程被“阻塞”在内存不足的状态。
        
5. **业界现状与结论**
    
    - 从系统整体资源利用率（吞吐量）来看，**全局置换**具有明显的优势，因此在实际的大多数操作系统中，**全局置换是更为常用的页面置换策略**。
        

## 9.5.4 非均匀内存访问

1. **NUMA 系统的定义与特点**
    
    - 传统的虚拟内存理论通常假设所有内存访问时间是相等的（UMA，均匀内存访问）。
        
    - **NUMA（非均匀内存访问）系统**中，由于多块系统板（主板）互连，**CPU 访问同一主板上的内存延迟小，访问其他主板内存延迟大**。这显著影响了系统性能。
        
2. **针对 NUMA 的内存分配与调度目标**
    
    - 核心目标是实现**内存和 CPU 的亲和性**：将物理帧尽可能分配给“靠近”运行进程的 CPU。
        
    - “靠近”指的是物理距离和硬件互连导致的**最小延迟**（通常意味着在同一系统板上）。
        
3. **算法层面的修改策略**
    
    - **调度策略协同**：调度程序需要记录每个进程**上一次运行的 CPU**，并优先将其再次调度到同一个 CPU 上。
        
    - **内存分配协同**：内存管理系统在分配帧时，优先分配靠近该 CPU 所在的系统板的物理内存。
        
    - **协同效果**：这种“调度 + 分配”的组合可以显著**提高 CPU 缓存命中率**，并**缩短内存访问时间**。
        
4. **多线程场景下的复杂性与 Solaris 的解决方案**
    
    - **挑战**：多线程进程可能同时运行在不同系统板的多个 CPU 上，单一的内存亲和性策略将失效。
        
    - **Solaris 的方案（Lgroup）**：
        
        - 将物理上相近的 CPU 和内存划分为一个 **延迟组（lgroup）**，并形成层次结构（基于组间延迟）。
            
        - 分配策略：尽量**将一个进程的所有线程调度在同一个 lgroup 内**，并在此 lgroup 中分配其所有物理内存。
            
        - 后备策略：若无法做到，则退而求其次，使用**邻近的 lgroup** 来分配剩余资源。
            
        - 终极目标：**最小化总体内存延迟，最大化 CPU 缓存命中率**。
