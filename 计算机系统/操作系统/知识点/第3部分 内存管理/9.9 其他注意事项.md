# 9.9 其他注意事项

本节聚焦于**其他注意事项**，是[[第九章 虚拟内存管理]]中的独立知识节点。

## 9.9.1 预调页面

> [!important] 预调页面的产生动机
> 
> - **纯请求调页的痛点**：当一个进程刚启动，或被换出后重新调入内存时，由于工作集（局部性页面）初始都不在物理内存中，会触发大量的、连续的缺页错误，导致启动阶段性能低下。
> - **预调页面的目的**：通过在进程执行前或恢复前，**一次性将所需的整个工作集（或所需页面）全部调入内存**，以避免启动时缺页错误的风暴。

> [!tip] 与工作集模型的结合
> 
> - 在采用工作集模型的系统中，操作系统会记录并保存每个进程的工作集信息。
> - 当进程因等待 I/O 或内存不足被挂起（换出）时，系统保留其工作集列表。
> - 当进程重新激活（换入）时，操作系统会在进程恢复执行之前，**自动将工作集内的所有页面提前调入内存**。

> [!note] 性能的权衡（成本效益分析）
> 
> 预调页面的核心问题是：**提前进行 I/O 调页的成本，是否小于由缺页错误逐页调入的成本？** 如果提前调入的页面最终并没有被进程使用，那么这些提前的磁盘 I/O 操作就是纯粹的资源浪费。

> [!example] 成功率数学分析
> 
> 设预调页面总数为 $s$，其中被实际使用到的页面比例（局部命中率）为 $\alpha$（$0 \le \alpha \le 1$）。
> 
> - **预调节省的缺页处理开销** = 实际有用的预调页面数 $\times$ 单次缺页处理成本 = $s \times \alpha$
> - **预调浪费的 I/O 开销** = 预调入但未使用的页面数 $\times$ 单次 I/O 成本 = $s \times (1 - \alpha)$
> 
> **判断标准**：
> - $\alpha \to 0$（接近 0）：绝大部分预调的页面都没用到，属于无效预调，方法**失败**。
> - $\alpha \to 1$（接近 1）：所有预调的页面都被用到，有效避免了多次缺页，方法**成功**。
            

## 9.9.2 页面大小

> [!info] 页面大小的范围与限制
> 
> - 页面大小总是 2 的幂，常见的范围是从 $2^{12}$ (4096 字节) 到 $2^{22}$ (4\ 194\ 304 字节 / $4\text{MB}$)。
> - 实际上**不存在唯一"最优"的页面大小**，选择时需要在多项相互冲突的因素之间进行权衡。

> [!tip] 倾向于"大页面"的优势
> 
> - **减小页表占用**：对于固定的虚拟内存空间，页面越大，需要的页表项越少，从而节省页表占用的内存开销。
> - **I/O 效率高**：磁盘 I/O 中，**寻道**（如 $20\text{ms}$）和**延迟**（如 $8\text{ms}$）的机械开销占据绝对主导地位。读取一个 1024 字节的大页面（$28.4\text{ms}$）远比读取两个 512 字节的小页面（合计 $56.4\text{ms}$）要快得多。
> - **减少缺页错误次数**：大页面能大幅减少因单次访问引起的缺页中断次数，从而降低了系统处理缺页中断（保存寄存器、排队 I/O、更新页表）的巨大开销。

> [!warning] 倾向于"小页面"的优势
> 
> - **减少内部碎片**：进程分配到最后时往往不能填满整个页面，小页面能显著降低这种内存浪费（如 512 字节页面浪费 256 字节，而 8192 字节页面浪费 4096 字节）。
> - **更好的局部性（精度）**：小页面能更精确地匹配程序的局部性，避免调入大量进程当前并不需要的无用数据，从而减少物理内存的浪费和无效的 I/O 传输。

> [!note] 历史演变趋势
> 
> 由于上述因素相互制约，页面大小的设计是一个典型的权衡过程。随着计算机硬件的发展，系统（即使是移动设备）的**页面大小呈逐渐增大的趋势**。从 1983 年《操作系统概念》第一版采用的 4096 字节，到现代系统，实际支持的页面大小已大幅提升。
        

## 9.9.3 TLB范围

> [!important] TLB 范围（TLB Reach）的定义与重要性
> 
> - **计算公式**：TLB 范围 = $\text{TLB 条目数} \times \text{页面大小}$
> - **核心意义**：它表示在 TLB（快表）不缺失的情况下，CPU 能够直接访问的物理内存总量。理想状态下，一个进程的**工作集（Working Set）应完全容纳在 TLB 范围内**，否则将频繁发生 TLB 缺失（TLB Miss），CPU 必须去查询慢速的页表，严重影响地址转换的效率。

> [!note] 扩大 TLB 范围的两种思路
> 
> | 方案 | 优点 | 缺点 |
> |------|------|------|
> | **增加 TLB 条目数** | 成倍增加范围 | TLB 采用昂贵的关联内存实现，成本高且功耗大，硬件上难以大规模扩展 |
> | **增大页面大小** | 迅速扩大范围（8KB→32KB 可翻四倍） | 导致明显的内部碎片和内存浪费 |

> [!example] 多页面大小机制的实践（以 UltraSparc / Solaris 为例）
> 
> - 现代处理器（如 UltraSparc）支持多种大小的页面（如 8KB、64KB、512KB、4MB）。
> - **基础页面**：使用 **8KB** 页面满足绝大多数普通应用程序的需求，以保持内存利用率。
> - **大页面**：使用 **4MB** 的大页面来映射**内核代码和数据**（共占 8MB），并允许数据库等大型内存密集型应用使用大页面。
> - **效果**：使系统的 TLB 范围跨度极大（Solaris 在 64 条目下，可覆盖从 512KB 到 256MB）。

> [!tip] 软件管理 TLB 的新趋势
> 
> - 支持多页面大小意味着**操作系统必须接管 TLB 的管理权**（硬件通常只支持单一固定大小的页面）。TLB 条目需要增加一个字段来标记页面的大小。
> - 虽然**软件管理 TLB**在单次操作上比硬件管理略慢，但通过使用多页面大小来提升 TLB 命中率和覆盖范围所获得的整体性能收益，远大于软件管理的开销。
> - 因此，**由操作系统动态控制混合页面大小**已成为现代计算机体系结构的发展趋势。
        

## 9.9.4 页表倒置

> [!important] 核心目的与设计动机
> 
> 倒置页表的设计初衷是为了**大幅减少用于维护虚拟地址到物理地址映射关系的物理内存开销**。传统的页表大小受限于进程的虚拟地址空间，而倒置页表的大小受限于物理内存的大小。

> [!tip] 核心机制与索引方式
> 
> - 系统在物理内存中仅维护一张全局的**倒置页表**。
> - 该表为物理内存中的**每一个物理帧**设置一个条目，而不是为每一个虚拟页设置条目。
> - 索引依据是**〈进程 ID，页码〉**配对，通过硬件或软件快速哈希查找，能迅速定位某个虚拟页当前是否在内存中，并对应的物理帧号。

> [!note] 优点与缺点
> 
> | 方面 | 说明 |
> |------|------|
> | **优点** | 极大节省了内存中用于保存页表映射信息的内存 |
> | **缺点** | 倒置页表仅包含当前已在物理内存中的页面信息，**丢弃了进程完整的虚拟地址空间映射信息**。这导致在处理缺页错误时，无法直接从倒置页表中获取换出页面的磁盘位置 |

> [!info] 解决方案：外部页表
> 
> - 为了弥补上述缺点，每个进程必须维护一个完整的**外部页表**（External Page Table）。
> - 该表本质上就是传统页表，记录了**进程每个虚拟页面在磁盘/交换空间上的具体位置**。它的优势在于**不需要常驻内存**，仅在发生缺页错误时才会被系统读取。

> [!warning] 特殊挑战：嵌套缺页错误
> 
> - 外部页表自身也有可能被换出到外存（磁盘）。
> - **关键难点**：当一个进程发生缺页错误时，操作系统需要先查阅外部页表来查找所缺页面的磁盘地址。但如果**外部页表本身此刻也不在内存中**，就会引发**第二次>缺页错误（嵌套缺页）**，要求操作系统将外部页表从磁盘调入内存。
>       
>    - **系统要求**：这种“缺页过程又触发缺页”的特殊情况，要求虚拟内存管理器和内核具备**递归处理缺失**的能力，并能精确处理由此带来的延迟。
        

## 9.9.5 程序结构

1. **程序局部性的重要性**
    
    - 请求调页对程序员是透明的，但编写具备良好**局部性（Locality）**的代码对提升虚拟内存性能至关重要。不合理的代码结构会导致极高的缺页错误率。
        
2. **C语言二维数组访问顺序的陷阱**
    
    - 在 C 语言中，二维数组是按**行优先**（Row-Major）存储在内存中的。
        
    - **反例（低效）**：外层循环按列遍历 `j`，内层循环按行遍历 `i`。访问内存是跳跃式的（先访问 `data[0][0]`，再访问 `data[1][0]`，这不在同一个物理页内），导致每次访问都可能触发一次缺页。
        
        int i, j;  
        int[128][128] data;  
          
        for (j = 0; j < 128; j++)  
          for (i = 0; i < 128; i++)  
              data[i][j] = 0;
        
    - **正例（高效）**：外层循环按行遍历 `i`，内层循环按列遍历 `j`。访问内存是连续的，能够顺序填满当前页面的所有字节，再切换下一页面。
        
        int i, j;  
        int[128][128] data;  
          
        for (i = 0; i < 128; i++)  
          for (j = 0; j < 128; j++)  
              data[i][j] = 0;
        
3. **缺页错误的数量对比**
    
    - **低效代码**：若分配的帧数少于行数（< 128），每次内存访问跨越不同页面，会产生高达 $128 \times 128 = 16\ 384$ 次缺页错误。
        
    - **高效代码**：按行连续访问，最坏情况也只需 **128 次**缺页错误（每行一个页面）。性能差距达到 100 倍以上。
        
4. **数据结构的选择**
    
    - 良好的局部性可以降低缺页错误率和工作集大小。
        
    - **堆栈（Stack）**具有优良的局部性（访问集中在顶部）。
        
    - **哈希表（Hash Table）**的局部性较差（访问是离散的、随机的）。
        
5. **编译器和加载器的优化**
    
    - **代码与数据分离**：将代码段设计为只读（重入代码），置换时无需写回磁盘（干净页面直接丢弃即可），降低 I/O 开销。
        
    - **页面边界对齐**：加载器应尽量将程序代码布置在**单一页面**内，避免跨越页面边界。
        
    - **分页打包**：将频繁互相调用的代码模块打包到同一个页面中，以最大化页面间引用的局部性。
        

## 9.9.6 I/O联锁与页面绑定

1. **问题起因：I/O 与页面置换的冲突**
    
    - 当进程发起 DMA（直接内存访问）I/O 请求时，设备控制器记录了缓冲区的**物理内存地址**。
        
    - 如果该进程在 I/O 完成前被换出（或发生缺页错误），采用**全局置换算法**可能会将该进程的 I/O 缓冲区页面置换到磁盘，并分配给其他进程。
        
    - 当 I/O 设备完成传输时，它会将数据直接写入之前记录的物理内存地址，而此时该地址已被其他进程占用，从而导致**数据被覆盖或损坏**。
        
2. **解决方案一：间接 I/O（代价高）**
    
    - 不直接在“用户内存”和“设备”之间进行 I/O。
        
    - 改为：先将数据从**用户内存复制到系统内核缓冲区**，然后在内核缓冲区和设备之间进行 I/O。
        
    - **缺点**：多了一次内存复制操作，带来了无法忽视的性能开销。
        
3. **解决方案二：页面锁定（Pinning）**
    
    - 为每个物理帧添加一个**锁定位**。
        
    - 进行 I/O 时，将涉及到的页面帧**锁定**在内存中，并禁止其被置换算法选中。
        
    - I/O 完成后，再将其**解锁**。
        
4. **锁定位的常见应用场景**
    
    - **内核内存**：操作系统的核心代码和数据通常被永久锁定，因为内核不允许发生缺页错误（会导致系统崩溃）。
        
    - **数据库管理**：数据库系统完全掌握自身的缓存管理，为了获得最佳性能，会锁定部分内存块以确保 I/O 的原子性和稳定性。
        
5. **防止刚调入的页面被立即置换**
    
    - 在页面置换中，存在一种不公平的情况：刚调入内存的页面（属于低优先级进程）被高优先级进程的缺页错误置换掉，导致低优先级进程前功尽弃。
        
    - 可以**暂用锁定位**来保护刚调入的页面，强制其至少被使用一次后，才允许被置换。
        
6. **风险与限制**
    
    - **危险隐患**：如果程序或操作系统发生错误，导致锁定位未被及时解锁，这些物理帧将**永久丢失（不可用）**。
        
    - **权限管控**：为防止用户滥用资源，系统通常禁止无特权的应用程序随意锁定大量内存。例如 Solaris 系统会提供加锁“提示”，但在资源紧张时（如空闲帧过少或锁定申请过量）有权**拒绝或忽略**这些提示请求。
