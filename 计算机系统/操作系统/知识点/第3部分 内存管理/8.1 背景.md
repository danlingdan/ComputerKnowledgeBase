# 8.1 背景

本节聚焦于**背景**，是[[第八章 内存管理策略]]中的独立知识节点。

## 8.1.1 基本硬件

> [!info] 存储访问层级与速度差异
> 
> - **CPU 可直接访问的设备**：仅限内存和寄存器。磁盘地址不能作为指令参数，数据需先加载到内存。
> - **速度对比**：寄存器访问耗时通常为 **1 个 CPU 时钟周期**；内存访问可能需**多个 CPU 时钟周期**。
> - **性能问题与优化**：内存访问过慢会导致 CPU 暂停（stall）。为缓解此问题，引入了**高速缓存（Cache）**，由硬件自动管理以加速访问，无需操作系统干预。
>

> [!important] 内存保护机制（基于基址与界限寄存器）
> 
> **目的**：防止用户进程非法访问操作系统内存或其他用户内存，实现进程隔离。
> 
> **实现原理**：使用两个专门寄存器：
> - **基地址寄存器（Base Register）**：存放进程合法物理内存的起始地址。
> - **界限地址寄存器（Limit Register）**：存放进程合法内存空间的长度（范围大小）。
> 
> ```mermaid
> graph TD
>     subgraph CPU

>     end
>     subgraph MMU

>     
>     style OS fill:#9E9E9E,color:white

>     style P2 fill:#FF9800,color:white
> ```
> 
> **检查机制**：CPU 硬件自动比较产生的地址：
> 1. 检查 **地址 ≥ 基地址**。
> 2. 检查 **地址 < 基地址 + 界限地址**。
> 3. 若越界，则触发"寻址错误"陷阱，陷入操作系统进行处理。
            
3. **权限与上下文切换**
    
    - **特权指令：** 设置基址和界限寄存器的指令是**特权指令**，只能在**内核模式**下执行，普通用户进程无法修改。
        
    - **内核特权：** 操作系统在内核模式下可无限制地访问内存，这允许其执行加载程序、处理 I/O、系统调用等操作。
        
    - **上下文切换：** 多任务系统中，操作系统会负责保存当前进程的寄存器状态，并从内存中恢复下一个进程的上下文到寄存器。
        

## 8.1.2 地址绑定

1. **程序执行与地址绑定定义**
    
    - 进程通常以**二进制可执行文件**形式存放在磁盘上的输入队列中。执行时需调入物理内存，并在过程中在磁盘与内存间移动。
        
    - **地址绑定**是指将程序中的符号地址（如变量名 `count`）逐步映射到物理内存绝对地址的过程。
        
> [!note] 地址绑定的三种时机
> 
> | 时机 | 描述 | 优点 | 缺点 |
> |------|------|------|------|
> | **编译时绑定** | 编译阶段确定物理地址，生成**绝对代码** | 无运行时开销 | 内存地址变动需重新编译 |
> | **加载时绑定** | 程序加载时完成绝对地址绑定，生成**可重定位代码** | 内存位置变动只需重新加载 | 运行时无法移动 |
> | **执行时绑定** | 运行时动态进行地址绑定 | 支持多道程序设计，进程可移动 | 需要 MMU 硬件支持 |
>

>     E --> F[Loader] --> G[Memory Image]

> ```
        
4. **内存管理的动态性**
    
    - 现代操作系统允许进程在物理内存中**任意位置驻留**（不仅仅从地址 00000 开始），这使得系统可以支持多道程序设计并实现内存的灵活利用。
        

## 8.1.3 逻辑地址空间与物理地址空间

1. **地址与空间定义**
    
    - **逻辑地址（CPU 视角）**：由 CPU 生成的地址。在“执行时绑定”模式下，也被称为**虚拟地址**。
        
    - **物理地址（内存视角）**：内存单元看到的实际地址（加载到内存地址寄存器中的值）。
        
    - **逻辑地址空间**：由程序所生成的所有逻辑地址的集合。
        
    - **物理地址空间**：上述逻辑地址所对应的所有物理地址的集合。
        
2. **不同绑定方式的地址关系**
    
    - 在**编译时或加载时绑定**中：逻辑地址 = 物理地址。
        
    - 在**执行时绑定**中：逻辑地址 \neq 物理地址，此时逻辑地址空间与物理地址空间相分离。
        
> [!important] MMU 与重定位寄存器（动态重定位机制）
> 
> **内存管理单元（MMU）** 是负责将虚拟地址映射为物理地址的硬件设备。
> 
> **重定位寄存器**：是基址寄存器的推广。用户进程生成的逻辑地址在发送到内存之前，会统一加上重定位寄存器的值。
> 
> **运行机制**：物理地址 = 逻辑地址 + 重定位寄存器值（示例：逻辑地址 346 + 重定位寄存器 14000 = 物理地址 14346）。
> 
> ```mermaid
> graph TD
>     CPU[CPU] -->|LA 346| MMU[MMU]
>     RR[Relocation Reg<br/>14000] --> MMU
>     MMU -->|PA 14346| MEM[(Memory)]
>     
>     style CPU fill:#FF6B6B,color:white
>     style MMU fill:#4CAF50,color:white
>     style MEM fill:#9E9E9E,color:white
> ```
        
4. **用户程序视角与内存隔离**
    
    - 用户程序**完全看不到**真实的物理地址。程序的指针、比较等操作均基于逻辑地址（如 346）进行。
        
    - 逻辑地址的物理重定位仅在**访问内存的瞬间**（如间接加载和保存）发生。
        
    - **地址空间分离的重要性**：逻辑地址范围仅从 $0 \sim \text{max}$（用户视角），而实际对应的物理地址范围是 $R + 0 \sim R + \text{max}$。这种"逻辑空间绑定到独立物理空间"的概念是现代内存管理系统的核心，也是实现进程隔离与内存保护的基础。
        

## 8.1.4 动态加载

1. **核心定义与目的**
    
    - **目的**：旨在克服早期“进程大小受限于物理内存大小”的问题，提高内存空间的利用率。
        
    - **定义**：动态加载是一种内存管理技术，即**一个程序（或代码段）只有在被实际调用时才会被加载到物理内存中**。
        
2. **运行机制**
    
    - 程序以**可重定位加载格式**保存在磁盘上。
        
    - 初始仅加载**主程序**进入内存执行。
        
    - 当需要调用子程序时，先检查内存中是否已存在。如果不存在，则由可重定位链接程序**动态加载**该程序，并**更新地址表**，随后将控制权交给新程序。
        
3. **核心优势**
    
    - **按需加载**：程序只有在“真正需要时”才占用内存。
        
    - **特别适用场景**：对于**错误处理、异常情况**等低频使用的代码块非常有效。即使程序整体文件很大，但在实际运行中可能只用到了极少部分代码，动态加载能大幅节省内存。
        
4. **系统与用户的责任**
    
    - **操作系统**：**不需要**提供特殊的底层操作系统支持（属于软件层面的优化策略，但操作系统可以提供动态加载的程序库来辅助程序员）。
        
    - **用户/程序员**：**负责任务主体**，需要程序员主动将程序设计成符合动态加载逻辑的模块化结构，才能发挥其优势。
        

## 8.1.5 动态链接与共享库

1. **定义与对比**：
    
    - **动态链接**：将库与程序的**链接过程推迟到运行时**进行。区别于静态链接（程序在加载或编译时就将系统库合并到二进制映像中）。
        
    - **核心价值**：避免了每个程序在磁盘和内存中保留一份独立的库副本，极大地**节省了磁盘空间和物理内存空间**。
        
2. **工作机制（存根机制）**：
    
    - 程序在二进制映像中保留了对应库函数的**存根（stub）**代码。
        
    - **首次执行**：存根检查库是否在内存，若不在则加载。随后，存根会将自身替换为库的**实际内存地址**，并开始执行。
        
    - **后续执行**：直接跳转到该地址执行，不再有链接开销。
        
3. **版本管理与更新**：
    
    - 支持库的**无缝更新**（如修复 Bug）。库可以替换为新版本，所有使用该库的程序将自动受益（无需重新编译/链接）。
        
    - 利用**版本号机制**（次要更改保留版本号，主要更改增加版本号）实现兼容性控制，允许多个版本的库共存。旧的程序可以继续使用旧版库，防止被不兼容的新版库破坏。
        
4. **系统支持**：
    
    - 与“动态加载”（用户程序自主控制）不同，**动态链接通常需要操作系统的底层支持**。
        
    - 操作系统需介入进程隔离保护，负责检查库是否在进程空间内，以及安全地允许**多个进程共享同一个内存地址空间**中的库代码。
