# 8.5 分页

本节聚焦于**分页**，是[[第八章 内存管理策略]]中的独立知识节点。

## 8.5.1 基本方法

**1. 基本概念与内存模型**

- **地址空间分离**：分页将**逻辑内存**划分为相同大小的**页（Page）**，将**物理内存**划分为同样大小的**帧（Frame）**。这使得逻辑地址空间与物理地址空间完全解耦（例如，32位系统的逻辑地址空间最高为4GB，但实际物理内存可以大于这个寻址范围）。
    
- **内存分配**：进程的页按需分配到任意可用的物理帧中，由于不需要物理连续，彻底解决了**外部碎片**问题。
    

> [!important] 地址转换与硬件支持
> 
> ```mermaid
> graph TD
>     CPU[CPU] -->|VA p=2, d=100| MMU[MMU]
>     PT["Page Table<br/>Page0->Frame5<br/>Page1->Frame3<br/>Page2->Frame7"] --> MMU
>     MMU -->|PA Frame7+d=100| MEM[(Memory)]
>     
>     style CPU fill:#FF6B6B,color:white
>     style PT fill:#4CAF50,color:white
>     style MEM fill:#9E9E9E,color:white
> ```
> 
> - **逻辑地址结构**：CPU生成的逻辑地址被切分为两部分：**页码（p, Page Number）** 和 **页偏移（d, Page Offset）**。选定页大小为 2^n 字节，逻辑空间为 2^m，则二进制中高 m-n 位为页码，低 n 位为偏移。
> - **转换机制**：页码 p 作为**页表**（Page Table）的索引，查询得到该页对应的物理帧号，然后将帧号与页偏移拼接，形成最终的物理地址。
> - **动态重定位**：分页机制本质上是**动态重定位**的一种形式（类似于每个页都有一个重定位寄存器），绑定的过程由硬件MMU完成，对程序员完全透明。
    

> [!note] 碎片问题（外部 vs 内部）
> 
> **优势：无外部碎片**。由于物理内存以固定大小的帧为单位进行分配，进程可以分散在不同帧中，不存在外部碎片。
> 
> **劣势：内部碎片**。分配内存以"帧"为单位，会导致进程最后一个页面无法完全填满。平均内部碎片大小为**半个页面**。在最坏情况下（需求大小 ≈ n × 页大小 + 1 字节），会产生几乎整个帧的内部碎片。
> 
> ```mermaid
> graph TD
>     subgraph Internal_Fragmentation
>         P1[Process 1<br/>3500 bytes used]
>         P2[Process 2<br/>2100 bytes used]
>         F1[Page Frame 1<br/>4KB]
>         F2[Page Frame 2<br/>4KB]
>         F1 --> P1
>         F2 --> P2
>         style F1 fill:#FF9800,color:white
>         style F2 fill:#FF9800,color:white
>     end
>     subgraph External_Fragmentation
>         MEM[Physical Memory]
>         MEM -->|Process A| A[500KB]
>         MEM -->|Fragment| F[50KB]
>         MEM -->|Process B| B[300KB]
>         MEM -->|Fragment| F2[30KB]
>         MEM -->|Process C| C[200KB]
>         style F fill:#f44336,color:white
>         style F2 fill:#f44336,color:white
>     end
> ```
    

> [!tip] 页面大小选择权衡
> 
> | 页面大小 | 优点 | 缺点 |
> |----------|------|------|
> | **小页面** | 内部碎片少，内存利用率高 | 页表项数量巨大，磁盘 I/O 效率低 |
> | **大页面** | 页表项少，磁盘 I/O 效率高 | 内部碎片增多 |
> 
> **现代趋势**：当前常见页大小为 $4\text{KB} \sim 8\text{KB}$，部分系统支持多种页大小（如 Solaris 支持 $8\text{KB}$ 和 $4\text{MB}$）。
    

**5. 操作系统管理职责**

- **帧表（Frame Table）**：操作系统必须维护一个数据结构，精确记录每个物理帧是空闲还是被某个进程的某个页占用。
    
- **页表副本与上下文切换**：内核为每一个进程维护一份页表。由于页表的切换是上下文切换的一部分，每次进程调度都需要替换硬件页表，因此**分页机制会增加上下文切换的开销**。
    
- **系统调用的物理地址映射**：当用户进程发起系统调用（如 I/O 操作）并传入逻辑地址作为参数时，操作系统内核必须首先通过页表将该逻辑地址转换为正确的**物理地址**，以确保硬件操作能够指向正确的内存缓冲。
    

## 8.5.2 硬件支持

1. **页表的硬件实现变迁与双重内存访问问题**
    
    - **早期实现**：使用**专用高速寄存器**存储页表（例如 DEC PDP-11，8个条目），地址转换极快。但寄存器数量少，无法支持现代大型页表（百万条目）。
        
    - **现代实现**：将页表存放在**物理内存**中，CPU 使用 **页表基址寄存器（PTBR）** 指向其位置。此法大大简化了上下文切换（只需切换 PTBR）。
        
    - **性能瓶颈（致命缺陷）**：由于页表在内存中，每次访问数据都需要 **两次内存访问**（第一次查页表得到物理帧号，第二次才是真正存取数据），导致内存访问速度减半。
        
> [!important] TLB（转换表缓冲区）机制与作用
> 
> **核心方案**：引入一个专用的、容量小但速度极快的高速硬件缓存（关联内存），称为 **TLB**，用于缓存最近访问的页表条目（页码-帧号对）。
> 
> ```mermaid
> graph TD
>     CPU[CPU] -->|虚拟地址| TLB{TLB查询}
>     TLB -->|命中| PA[物理地址]
>     TLB -->|未命中| PT[页表查询]
>     PT -->|更新TLB| TLB
>     PT --> PA
>     PA --> MEM[(物理内存)]
>     
>     style TLB fill:#FF6B6B,color:white
>     style PT fill:#4CAF50,color:white
>     style MEM fill:#9E9E9E,color:white
> ```
> 
> **工作流程**：先查 TLB，若命中（找到映射），则直接得到物理地址，且此过程通常是 CPU 流水线的一部分，**基本无额外性能损耗**。若**未命中（TLB miss）**，则需访问内存中的页表获取信息，并**将此条目回填至 TLB**。
        
3. **命中率与有效内存访问时间（EAT）**
    
    - **命中率（Hit Ratio）**：CPU 生成的虚拟地址在 TLB 中找到映射的概率。
        
    - **性能计算示例**：假设一次内存访问需 $100\text{ns}$，TLB 查询无额外延迟。
        
        - 若命中率 80%： $\text{EAT} = 0.80 \times 100 + 0.20 \times 200 = 120\text{ns}$，性能损耗 20%。
            
        - 若命中率 99%： $\text{EAT} = 0.99 \times 100 + 0.01 \times 200 = 101\text{ns}$，仅损耗 1%。这说明高命中率对于分页性能至关重要。
            
4. **安全性与多进程保护（ASID）**
    
    - **ASID（地址空间标识符）**：将每个 TLB 条目与特定的进程 ID 绑定。
        
    - **优势**：防止不同进程在发生上下文切换时，错误地使用前一个进程残留的 TLB 缓存映射（这会导致访问到错误的内存地址）。
        
    - **无 ASID 的代价**：若缺乏 ASID，上下文切换时 **必须刷新（Flush）** 整个 TLB，这会导致大量缓存失效，降低性能。
        
5. **现代硬件发展（多级 TLB）与操作系统设计的联动**
    
    - **多级 TLB**：现代 CPU（如 Intel Core i7）通常包含 **L1 TLB（指令和数据分离）** 和 **L2 TLB**。L1 未命中时会去检查 L2，L2 未命中才会去查内存中的页表（需数百个时钟周期，代价巨大）。
        
    - **软硬协同**：TLB 本质是硬件，但操作系统设计者必须深入了解具体平台（如不同代际的 Intel CPU）的 **TLB 特性**来调整分页实现和优化策略，以达到最佳性能。这与现代计算机中多层高速缓存的设计趋势是一致的。
        

## 8.5.3 保护

**1. 页表内的访问权限保护（保护位）**

- **机制**：每个页表项中设有保护位（如一位或多位），用于定义页面的访问权限（如：**只读、可读写、只执行**）。
    
- **执行过程**：当 CPU 进行内存引用时，硬件在计算物理地址的同时会检查这些保护位。若发生违规操作（例如，试图向设置为“只读”的页面执行写入），硬件会触发**陷阱（Trap）**，将控制权交给操作系统进行处理。
    

**2. 逻辑地址空间的合法性检查（有效-无效位）**

- **有效-无效位（Valid-Invalid bit）**：用于指示某一页是否属于当前进程的逻辑地址空间，即“合法有效”的页。
    
- **执行过程**：如果 CPU 生成的逻辑地址对应的页表项该位为“无效”，则说明访问了非法的地址空间。操作系统会捕捉到这种**无效页引用**，并作相应处理（如报错终止进程）。
    
- **示例说明**：在 $2^{14}$ 字节（16KB）的地址空间中，若程序仅用到 0~10468 的位置，以 $2\text{KB}$ 为页面大小时，会引发页表后部分的页（页 6 和 7）有效-无效位置为无效。
    

**3. 分页内部碎片导致的地址空洞**

- 由于页面大小固定的限制，即使程序的实际逻辑地址只到 10468，但由于物理内存分配必须以页为单位，系统仍会将整页（包含未使用的 12288~16383 部分）映射为有效。这反映了分页机制存在**内部碎片**的问题。
    

**4. 针对内存浪费的优化（页表长度寄存器）**

- **背景问题**：进程通常只占用其地址空间的一小部分，如果为操作系统支持的整个最大地址空间都预先建立页表条目，会浪费大量的内存。
    
- **解决机制**：引入**页表长度寄存器（PTLR）**，它存放了当前进程的页表大小。在地址转换之前，硬件会利用 PTLR 检查逻辑地址是否超出当前进程的有效范围。如果越界，则会被操作系统捕捉，从而避免了为未被使用的地址建立无效页表条目所带来的内存浪费。
    

## 8.5.4 共享页

1. 分页技术的关键优势：共享公共代码
    

在分时系统或多用户环境中，如果多个进程需要执行同一份代码（例如文本编辑器、编译器、运行时库），分页机制允许这些进程在**物理内存中仅保留一份代码副本**。

2. 共享的前提条件：可重入代码 (Reentrant Code)
    

- **定义**：可重入代码或纯代码是指在**执行期间不会自我修改**的代码。
    
- **安全保证**：为了保证代码共享的安全，代码的**只读属性不能依赖程序员保证**，而必须由**操作系统强制实现**。操作系统通过设置页表项的保护位来防止任何进程对共享代码段进行写操作。
    

3. 内存节省的量化分析
    

教材通过一个编辑器实例清晰展示了共享页带来的巨大内存节省：

- **不共享的情况**：40 个用户每人一份代码与数据，总需求为：
    
    $$  
    $40 \times (150\text{KB} \text{代码} + 50\text{KB} \text{数据}) = 8000\text{KB}$  
    $$
    
      
    
- **共享代码的情况**：物理内存中仅保存一份编辑器代码，各用户独享数据空间：
    
    $$
    1 \times 150\text{KB} \text{代码} + 40 \times 50\text{KB} \text{数据} = 2150\text{KB}
    $$
    
      
    
    _节省了高达 $5850\text{KB}$ 的物理内存空间。_
    

4. 底层实现机制
    

- **页表映射分离**：通过逻辑地址到物理地址的映射，不同进程的页表项中，**代码部分的页码被映射到物理内存中相同的帧号**。
    
- **数据独占**：而各个进程的**数据页则是映射到物理内存中各自独立的帧**，从而保证了各进程的数据隔离。
    

5. 扩展应用领域
    

- **系统库与系统组件**：窗口系统、运行时库、数据库系统等高复用率的系统组件通常采用此机制。
    
- **进程间通信 (IPC)**：共享页机制也是操作系统实现**共享内存（Shared Memory）**通信方式的基础，它允许不同进程映射同一物理帧来进行高效的数据交换。
