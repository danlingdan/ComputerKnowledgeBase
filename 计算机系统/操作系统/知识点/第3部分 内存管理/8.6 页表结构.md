# 8.6 页表结构

本节聚焦于**页表结构**，是[[第八章 内存管理策略]]中的独立知识节点。

## 8.6.1 分页结构

1. **分层分页的引入背景**
    
    - **问题**：现代计算机支持极大的逻辑地址空间（如 32 位或 64 位），导致页表本身占用的内存极其巨大。
        
        - 例如：32位系统、4KB页面时，单进程的页表就需要 $4\text{MB}$ 的连续物理内存。
            
    - **解决方案**：采用**分层分页（多级页表）**，将巨大的页表本身再拆分为更小的、可以被分页管理的单元，避免在内存中连续占用大块空间。
        
> [!important] 两级分页结构（以 32 位系统为例）
> 
> **地址拆分**：逻辑地址被分为三个部分：[外部页号 p₁] [内部页号 p₂] [页面偏移 d]。
> 
> ```mermaid
> graph TD
>     CPU[CPU] -->|逻辑地址<br/>p₁=2, p₂=5, d=100| MMU[MMU]
>     PTBR[PTBR] --> PDT[页目录表]
>     PDT -->|p₁=2| PT[页表]
>     PT -->|p₂=5| FRAME[物理帧号]
>     FRAME -->|拼接偏移| PA[物理地址]
>     PA --> MEM[(物理内存)]
>     
>     style CPU fill:#FF6B6B,color:white
>     style PDT fill:#4CAF50,color:white
>     style PT fill:#FF9800,color:white
>     style MEM fill:#9E9E9E,color:white
> ```
> 
> **访问过程（向前映射）**：
> 1. 使用 p₁ 作为索引，查找**外部页表**（页目录表），得到页表的物理帧号。
> 2. 使用 p₂ 作为索引，在该页表的物理帧中查找到实际内存的物理帧号。
> 3. 将物理帧号与偏移量 d 结合，得到最终的物理地址。
        
3. **64 位系统的挑战与多级分页的局限性**
    
    - **巨大膨胀**：在 64 位系统中，如果仅使用两级分页，外部页表本身的容量可能高达 $16\text{GB}$（如文中计算示例：$2^{44}$ 字节）。为了继续压缩，只能引入三层、四层甚至更多层的分页（例如需继续将外部页表再次细分）。
        
    - **性能瓶颈**：解决容量问题会带来严重的**性能问题**。例如 UltraSPARC 架构需要多达 **7 级** 分页。每多一级分页，地址转换就多一次内存访问。7 级分页意味着每次内存访问都需要额外进行 7 次内存读取，这是不可接受的开销。
        
    - **结论**：虽然分层分页在 32 位环境下非常有效，但对于 **64 位地址空间**，多级分层页表被认为是**不适当的**，需要引入哈希页表或倒置页表等其他结构。
        

## 8.6.2 哈希页表

1. **定义与适用场景**
    
    - **核心定义**：哈希页表是一种利用哈希函数来加速大地址空间（尤其是大于 32 位）内存地址转换的技术。
        
    - **应用目的**：解决传统多级页表在超大地址空间（如 64 位）下因层数过多导致性能低下的问题。
        
> [!important] 数据结构与哈希冲突处理
> 
> **哈希表 + 链表**：哈希页表的结构是一个哈希数组，数组中的每个条目指向一个链表。
> 
> **处理冲突（拉链法）**：当不同的虚拟页码经过哈希函数计算后得到相同的索引（即发生哈希冲突）时，它们会通过链表连接在一起。
> 
> ```mermaid
> graph TD
>     subgraph HashPageTable
>         A[Hash Table]
>         A --> L1["List 1<br/>VPN1->Frame5<br/>VPN2->Frame8"]
>         A --> L2["List 2<br/>VPN3->Frame12"]
>         A --> L3["List 3<br/>VPN4->Frame3<br/>VPN5->Frame7<br/>VPN6->Frame15"]
>     end
>     style A fill:#FF6B6B,color:white
>     style L1 fill:#4CAF50,color:white
>     style L2 fill:#4CAF50,color:white
>     style L3 fill:#4CAF50,color:white
> ```

> 
> **链表元素结构**：每个节点包含三个关键字段：
> 1. **虚拟页码**（用于键值比对）
> 2. **映射的物理帧号**（即查找到的结果）
> 3. **指向下一个链表节点的指针**
    
3. **地址转换算法步骤**
    
    - 提取 CPU 产生的逻辑地址中的 **虚拟页码 p**。
        
    - 将 p 代入哈希函数，定位到哈希表中的某一个槽位。
        
    - 遍历该槽位对应的**链表**，依次比对链表节点的 **虚拟页码** 字段。
        
    - 如果匹配成功，读取该节点的 **物理帧号** 字段，结合逻辑地址的 **页偏移 d** 组成最终的物理地址。
        
    - 如果遍历完整个链表仍未匹配，则触发缺页异常或非法访问处理。
        
4. **针对 64 位系统的优化：聚簇页表**
    
    - **背景**：在 64 位系统下，传统哈希页表的链表可能依然很长，且内存引用往往不连续（稀疏地址空间）。
        
    - **优化方案（聚簇页表）**：在哈希表中，**一个条目**不再仅映射单一物理帧，而是映射 **多个连续的物理帧**（例如一次映射 16 页）。
        
    - **优势**：特别适合**稀疏（sparse）** 的 64 位地址空间。它能有效减少哈希表的大小，降低冲突概率，提升大型连续数据块的查询效率。
        

## 8.6.3 倒置页表

1. 传统页表的局限
    

传统的进程关联页表是根据**虚拟地址空间**来建立条目的。在 64 位系统下，单进程的页表可能占据数百万乃至数十亿个条目，这会导致极大的物理内存开销。

> [!important] 倒置页表（Inverted Page Table）的设计概念
> 
> ```mermaid
> graph TD
>     subgraph InvertedPageTable
>         direction LR
>         F0[Frame 0<br/>ASID=P1, VPN=5]
>         F1[Frame 1<br/>ASID=P2, VPN=3]
>         F2[Frame 2<br/>ASID=P1, VPN=8]
>         F3[Frame 3<br/>ASID=P3, VPN=2]
>         F4[Frame 4<br/>Free]
>     end
>     
>     style F0 fill:#4CAF50,color:white
>     style F1 fill:#4CAF50,color:white
>     style F2 fill:#4CAF50,color:white
>     style F3 fill:#4CAF50,color:white
>     style F4 fill:#9E9E9E,color:white
> ```
> 
> > [!note] 图例：Inverted Page Table = 按物理帧索引的统一页表
> 
> **存储反转**：倒置页表将映射关系"反转"，它是根据**物理内存的帧（Frame）**来建立条目的。
> 
> **唯一性**：**整个系统只有一个页表**，且该页表的大小与系统的物理内存大小成正比，而非与进程的逻辑地址空间大小成正比。
> 
> **表项组成**：每个物理帧在表中对应一个条目，条目主要存储两项关键信息：
> 1. 当前驻留在该物理帧中的**虚拟页号**。
> 2. 拥有该页的**进程 ID**（即地址空间标识符 ASID）。
        

3. 地址转换流程与公式
    

- **查找依据**：CPU 输出的虚拟地址被提取为 `<进程ID, 虚拟页号, 偏移>`。
    
- **搜索匹配**：内存系统在倒置页表中搜索 `(进程ID, 虚拟页号)` 这一对组合。
    
- **物理地址生成**：假设该组合在倒置页表的索引 **i** 处被匹配到（意味着物理帧 i 存放了该数据），则物理地址直接由 $(i, \text{偏移})$ 构成。若找不到匹配项，则产生非法地址访问。
    

4. 优缺点与挑战
    

- **优点（大幅节省空间）**：从根本上解决了页表占用海量连续物理内存的问题，特别适合物理内存相对较小的 64 位系统。应用架构包括 64 位 UltraSPARC 和 IBM PowerPC。
    
- **缺点 1：转换速度慢**。
    
    - 原因：页表按物理地址排序，必须遍历表寻找匹配的虚拟地址。
        
    - 解决方案：利用 **哈希表** 将搜索时间缩短为 O(1) 或接近常数时间（增加一次哈希表读操作，通常配合 TLB 进一步优化）。
        
- **缺点 2：共享内存实现困难**。
    
    - 原因：一个物理帧只能存一个 `(进程ID, 虚拟页)` 条目，而共享内存要求多个虚拟地址指向同一个物理帧。
        
    - 解决方案：只保存一个映射关系，当其他进程访问该共享内存时，会触发**页错误**，由操作系统在缺页异常处理程序中进行后续响应。
        

## 8.6.4 Oracle SPARC Solaris

1. **系统与内存管理**：Solaris 是运行于 SPARC 处理器上的真正 64 位操作系统。它通过多级页表来管理虚拟内存，并结合哈希表高效地完成地址映射，同时避免耗尽物理内存。
    
2. **哈希表架构**：系统设计了两个独立的哈希表（一个用于内核，一个用于用户进程）。其条目代表连续的虚拟内存区域（包含基地址及跨度），这种设计比仅映射单页的条目更为高效。
    
3. **缓存与加速机制**：
    
    - **TLB (Translation Lookaside Buffer)**：位于 CPU 内部，用于保存转换表条目（TTE），以实现极速的硬件级地址查找。
        
    - **TSB (Translation Storage Buffer)**：内存中的缓存区，用于存放最近访问页面的 TTE。
        
4. **地址转换（TLB Walk）流程**：
    
    - 访问内存时，硬件优先搜索 TLB 进行转换。
        
    - 若 TLB 未命中，硬件自动搜索内存中的 TSB。
        
    - 若 TSB 匹配成功，将对应条目复制到 TLB 并完成转换。
        
    - 若 TSB 也未命中，则触发内核中断，系统从对应哈希表中查找并创建新的 TTE，保存到 TSB 中。随后，CPU 的内存管理单元（MMU）自动将 TTE 加载到 TLB。最后中断返回，MMU 完成地址转换并获取内存数据。
