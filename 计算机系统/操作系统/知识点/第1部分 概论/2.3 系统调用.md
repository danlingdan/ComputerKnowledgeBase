# 2.3 系统调用

本节聚焦于**系统调用**，是[[第二章 操作系统结构]]中的独立知识节点。

> [!summary] 一句话定义
> 系统调用（System Call）是操作系统提供给**用户程序**的服务接口，是用户程序请求内核执行特权操作（如读写文件、创建进程、网络通信）的唯一途径。

> [!example] 实际场景验证
> 执行"复制文件"需要涉及：获取文件名 → 打开源文件 → 创建目标文件 → 循环读写 → 错误判断处理 → 关闭文件 → 正常退出，这些步骤由多个底层系统调用串联完成。

## 为什么程序员主要用 API 而不是直接调用系统调用？

> [!reason] 核心原因
> - **易用性和细节屏蔽**：直接使用系统调用非常复杂，关注底层细节。
> - **可移植性（重点）**：API 是操作系统层面的**通用契约**。基于标准化 API（如 **Windows API、POSIX API、Java API**）编程，可以在不同底层操作系统上编译运行。
> - **隐藏实现细节**：例如，Windows 的 `CreateProcess()` API 后台映射到底层的 `NTCreateProcess()` 系统调用。

## 系统调用的执行流程与系统调用接口

![[assets/image-20260707020235235.png]]

> [!mechanism] 工作机制
> 系统调用接口（System-Call Interface）是夹在应用程序（API）和操作系统内核之间的**桥梁**：
> 1. 运行时库接管 API 调用，将系统调用映射为一个**唯一的数字编号**。
> 2. 接口通过编号查表（索引列表），精确跳转到内核中对应的实现函数。
> 3. 执行完毕后将结果和状态返回给应用程序。

## 参数传递机制

> [!problem] 核心挑战
> 系统调用往往需要传递大量参数（如文件路径、缓冲区长、权限位等），但内核无法直接访问用户程序的内存。

> [!solution] 三种传参方式
> - **方法一（直接寄存器）**：最简单，但受限于 CPU 寄存器数量。
> - **方法二（内存块/表地址）**：将参数存入内存块或表中，**仅将首地址存入寄存器**传给内核。**Linux、Solaris** 等系统采用此法。
> - **方法三（堆栈）**：程序将参数**压入（push）**堆栈，操作系统执行时**弹出（pop）**，不限制参数数量或长度。
