# 2.7 操作系统的结构

本节聚焦于**操作系统的结构**，是[[第二章 操作系统结构]]中的独立知识节点。

> [!summary] 四种经典结构
> 操作系统经历了从简单结构到分层、微内核、模块化，最终走向混合系统的演变过程。

## 2.7.1 简单结构

**MS-DOS：无结构、无保护的"原始"系统**

> [!danger] 致命缺陷
> **缺乏接口和层次的隔离**。应用程序可以直接调用底层 I/O 程序来操作显示器和磁盘硬件，导致系统极不稳定。

- **演变路径**：最初由于硬件资源极度有限，只为实现最小功能而设计，没有进行模块化划分。
- **后果**：一个用户程序的错误或恶意攻击就能轻易**导致整个操作系统崩溃**。这也是受限于当时硬件（Intel 8088 芯片不支持双模式硬件保护）的无奈之举。

**早期 UNIX：传统的"单片式（Monolithic）"结构**

> [!mechanism] 基本组成
> 将操作系统分为**内核（Kernel）**和**系统程序**两大部分。内核包含了从系统调用接口层到物理硬件之上的所有核心功能（文件系统、CPU调度、内存管理等）。

> [!comparison] 单片结构的优缺点
> - **缺点**：内核中集成了大量功能，导致系统**难以设计和实现**，增加了维护难度。
> - **优势**：所有核心功能集中在一个内核中，**系统调用接口与内核间的通信开销非常小**，从而提供了**极高的性能**。

**经典的权衡与现实选择**

> [!note] 现代应用
> 尽管单片结构在设计和维护上复杂，但由于其**卓越的性能优势**，现代主流操作系统（如 **UNIX、Linux 和 Windows**）的基础架构依然沿用了这种单片结构（不过现代版本通常会加入动态加载的模块化特性来缓解维护难题）。

## 2.7.2 分层方法

> [!mechanism] 核心架构定义
> 将操作系统划分为**若干层（级）**，从最底层的**硬件（层0）**到最高层的**用户接口（层N）**。**每一层只能调用其更低层**提供的功能和服务，形成单向下行依赖。

**主要优势**

- **简化构造与调试**：由于存在严格的依赖关系，系统可以实现**自底向上的逐步调试**（底层调试无误后，上层再接入）。
- **信息隐藏**：每层只需知道下层"做什么"，而**无需知道"怎么实现"**，实现了良好的模块化和封装。

**设计难点与缺陷**

> [!problem] 定义层级的困难
> 合理的层级划分需要非常仔细的规划。例如，备份存储设备驱动（虚拟内存）既需要放在内存管理之下（为了管理内存），又通常需要放在 CPU 调度之上（为了等待 I/O），导致产生循环依赖的矛盾。

> [!problem] 性能开销
> 由于多层调用，用户进程执行一次 I/O 可能需要层层穿透（I/O层 → 内存管理层 → CPU调度层 → 硬件），产生大量**额外开销**，导致系统调用速度比非分层系统慢很多。

**现代演进与权衡**

> [!tip] 当代折中方案
> 现代通用操作系统通常采用**"功能更多但层级更少"**的设计，既能保留分层法带来的**模块化**优势，又能大幅减少层间交互带来的**性能损耗**。

## 2.7.3 微内核

> [!mechanism] 核心理念
> 将传统操作系统内核中除**最基本功能**（最小的进程管理、内存管理、**进程间通信 IPC**）以外的所有组件，移出内核空间，作为**用户空间的独立服务/进程**运行。

> [!mechanism] 通信机制
> 用户程序（客户端）与用户空间的服务（如文件服务器）**不直接交互**，必须通过**消息传递（Message Passing）**，由微内核作为中转站。

**微内核架构的核心优势**

- **易扩展性**：在用户空间增加新服务无需修改内核。
- **高度可移植**：内核本身极小且与硬件无关，非常容易移植到不同的 CPU 架构。
- **高可靠性与安全性**：大多数服务和驱动运行在**用户模式**，相互隔离。**一个服务或驱动崩溃不会导致整个内核崩溃**。

**最大的缺陷与演进折中**

> [!problem] 性能开销
> 频繁的用户态/内核态切换以及多次的消息传递，会产生巨大的上下文切换开销，导致微内核结构的**系统性能（如 I/O 吞吐量）显著低于单片内核**。

> [!example] Windows 案例
> - **Windows NT**：最初采用微内核，但性能不佳。
> - **NT 4.0**：将部分关键组件移回内核空间以提高性能。
> - **Windows XP**：核心架构实际上更靠近**单片内核**。

> [!conclusion] 混合架构趋势
> 现代操作系统通常倾向于**混合架构**——在保留模块化和可扩展性的同时，为了追求极限性能，会将大部分核心子系统（如文件系统、网络栈）移回内核中，演变为**"改良的单片内核"**。

## 2.7.4 模块

> [!mechanism] 核心设计思想
> 操作系统内核只保留一个最小的**核心组件**（提供最基础的服务）。其他功能（文件系统、设备驱动、调度算法）以**模块**的形式存在，可以在内核**启动时或运行时动态加载**。

**模块化架构的核心优势**

- **无需重新编译**：增加新功能（如支持新文件系统）只需加载对应的模块，而无需重新编译整个内核。
- **极高的灵活性**：任何模块之间都可以相互调用，比严格分层的"分层方法"更加灵活。
- **性能优越（优于微内核）**：像微内核一样实现了模块化和核心精简，但**效率更高**。因为模块之间是直接进行**函数调用**，而不像微内核那样需要通过**消息传递**进行通信。

## 2.7.5 混合系统

> [!summary] 必然趋势
> 现实世界的操作系统很难只采用单一的理论结构。主流操作系统（如 Linux、Windows、Mac OS X）都融合了**单片内核**（追求高效）、**微内核**（便于扩展与安全）和**模块化**（支持动态加载）的特点，形成了混合结构。

**Mac OS X 架构（Mach微内核 + BSD 内核）**

> [!structure] 四层架构
> 1. **底层**：**Mach 微内核**（负责内存管理、IPC、线程调度等基础任务）。
> 2. **中层**：**BSD 内核**（提供网络、文件系统、POSIX 命令行接口）。
> 3. **驱动层**：**I/O Kit**（用于驱动开发）。
> 4. **顶层**：**Aqua UI** 和 **Cocoa** 开发框架。

**iOS 架构（Mac OS X 的移动化演变）**

> [!structure] 主要变化
> - 核心架构沿用了 Mac OS X 的底层，但针对移动设备进行了重新分层。
> - 顶层采用 **Cocoa Touch** 框架，专门支持**触摸屏**等移动设备特有的硬件交互。
> - 中间层包含**媒体服务**和**核心服务**（支持云服务、数据库等）。

**Android 架构（定制 Linux 内核 + Dalvik 虚拟机）**

> [!structure] 主要特色
> - **底层**：采用经过 Google **定制修改的 Linux 内核**，尤其新增了**电源管理**功能。
> - **运行环境**：早期采用 **Dalvik 虚拟机**，专门针对移动设备的**内存和处理能力受限**进行了深度优化。应用开发使用 Java，但底层通过 **Android API** 将 Java 字节码转换后运行于 Dalvik 之上。
