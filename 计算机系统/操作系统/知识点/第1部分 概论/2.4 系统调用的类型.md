# 2.4 系统调用的类型

本节聚焦于**系统调用的类型**，是[[第二章 操作系统结构]]中的独立知识节点。

## 2.4.1 进程控制

> [!summary] 核心职责
> 管理进程的生命周期（创建、执行、终止），并提供同步机制确保并发安全。

**进程的生命周期管理**

- **终止方式**：程序可以**正常终止** (`end()`) 或**异常终止** (`abort()`)。异常触发错误陷阱，操作系统**转储内存**以供**调试器**排查。
- **控制权流转**：程序结束后，控制权回到**命令解释程序**。

**程序加载与执行**

- 进程执行时可能需要**加载** (`load()`) 或**执行** (`execute()`) **另一个程序**。
- **核心问题**：原有程序是**被覆盖/丢失**、**被保存**，还是**与新程序并发执行**？这决定了底层系统调用的实现逻辑。

**核心系统调用 API 对比**

| 功能类别 | Windows API | UNIX API |
| --- | --- | --- |
| 进程创建/终止 | `CreateProcess()` / `ExitProcess()` | `fork()` / `exit()` / `wait()` |
| 进程通信 | `CreatePipe()` / `CreateFileMapping()` | `pipe()` / `shm_open()` |
| 权限管理 | 安全描述符 API | `chmod()` / `umask()` |

**进程同步与互斥**

> [!warning] 并发风险
> 多进程共享数据时，必须通过**锁定机制**确保数据完整性和一致性。

- **关键系统调用**：`acquire_lock()` 和 `release_lock()`，防止并发进程同时修改同一数据导致混乱。

**单任务系统 vs 多任务系统**

> [!comparison] MS-DOS（单任务）vs FreeBSD（多任务）
> - **MS-DOS**：不创建新进程。执行新程序时，**新程序直接覆盖命令解释程序的一部分**。程序结束后重新加载解释程序。这是一种"**覆盖式**"的极简执行机制。
> - **FreeBSD**：采用经典 UNIX 模型。通过 `fork()` **克隆**新进程，再通过 `exec()` 加载新程序。父进程可选择**等待（wait）**子进程完成，或在**后台**执行子进程。

## 2.4.2 文件管理

> [!summary] 核心职责
> 提供文件的生命周期管理、目录操作和属性管理。

**文件的生命周期管理**

- 基本操作：**创建（`create`）、打开（`open`）、读写（`read`/`write`）、重定位文件指针（`reposition`）、关闭（`close`）、删除（`delete`）**。

**目录与文件属性管理**

- 目录作为特殊文件，同样支持上述生命周期操作。
- 系统提供 `get_file_attributes` 和 `set_file_attributes` 获取和设置文件元信息（文件名、类型、保护权限、记账信息等）。

**更高层的文件操作**

- 部分操作系统提供**移动（`move`）**和**复制（`copy`）**等更高级的复合系统调用。

**API 实现方式的差异**

- **直接由内核系统调用提供**。
- **通过外部独立的系统程序（可执行工具）实现**。只要这些程序可以被其他程序直接调用，它们在功能和接口层面上就等同于系统 API。

## 2.4.3 设备维护

> [!summary] 核心职责
> 管理独占设备的分配与释放，并统一文件与设备的抽象接口。

**资源的独占与同步**

- 进程必须先**请求（`request()`）**并**获得**设备的独占使用权，用完后必须**释放（`release()`）**。
- 若管理不当，极易引发**设备争用**甚至**系统死锁**（详见第7章）。

**文件与设备的统一抽象**

> [!philosophy] UNIX 哲学："一切皆文件"
> 操作系统将 I/O 设备和普通文件视为同一种"文件-设备"结构。程序员可以使用**同一组系统调用**（`read()`、`write()`、`reposition()`）来操作文件或外部设备（磁盘、打印机、终端等），极大简化了应用程序开发。

**设备识别方式**

- 通过**特殊文件名**（如 `/dev/tty`）、**目录位置**或**文件属性**来区分具体的 I/O 设备。

## 2.4.4 信息维护

> [!summary] 核心职责
> 提供系统信息查询、调试支持和性能分析功能。

**时间与系统全局信息获取**

- 获取**当前时间和日期**（`time()`、`date()`）。
- 获取**系统状态数据**（登录用户数量、操作系统版本、内存/磁盘剩余容量等）。

**调试功能支持**

- **内存转储（`dump()`）**：保存程序当前内存状态，用于事后故障分析。
- **系统调用跟踪（Trace）**：记录程序执行过程中的所有系统调用序列。
- **单步执行（Single step）**：每执行一条机器指令触发一次**陷阱（中断）**，调试器捕获陷阱实现精细化跟踪。

**程序性能分析（Time Profile）**

> [!mechanism] 实现原理
> 操作系统通过**定期触发定时器中断**，在每次中断时记录**程序计数器（PC）**的当前值。统计这些记录的分布，生成程序的**时间曲线**，帮助开发者了解各部分代码的耗时。

**进程属性管理**

- 操作系统维护所有进程的元数据。
- 提供 `get_process_attributes()` 和 `set_process_attributes()` 接口，支持获取或修改进程属性（如调整优先级）。

## 2.4.5 通信

> [!summary] 两种通信模型
> 进程间通信主要采用**消息传递模型**和**共享内存模型**，各有优劣。

**消息传递模型（Message-Passing Model）**

> [!mechanism] 核心机制
> 进程通过**相互发送消息**交换信息，可直接发送或通过**共同邮箱**间接发送。

- **寻址与连接**：需要知道对方的标识（**主机名/IP地址**和**进程名**）。系统调用 `get_hostid()` / `get_processid()` 获取标识符，通过 `open_connection()` / `close_connection()` 建立或断开连接。
- **经典架构**：采用 **C/S（客户机/服务器）** 架构。服务器通常是**守护进程（Daemon）**，通过 `wait_for_connection()` 监听连接，接受连接后使用 `read_message()` / `write_message()` 进行数据交互。

**共享内存模型（Shared-Memory Model）**

> [!mechanism] 核心机制
> 通过 `shared_memory_create()` 创建共享内存，并用 `shared_memory_attach()` 将其映射到进程地址空间。

- **内存访问限制的解除**：操作系统通常禁止进程访问其他进程的内存，但共享内存要求**进程间达成一致，取消该限制**。
- **数据控制权**：数据的格式和含义完全由进程自己决定，**操作系统不参与控制**。
- **同步警告**：并发写入可能导致数据混乱，进程需要自己负责**同步**。**线程（thread）默认就是共享内存的**（详见第4章）。

**两种模型的对比与权衡**

| 特性 | 消息传递模型 | 共享内存模型 |
| --- | --- | --- |
| 数据量 | 适合少量数据 | 适合大量数据 |
| 同步复杂度 | 低（无需额外同步） | 高（需要锁机制） |
| 速度 | 较慢 | 极速（内存级传输） |
| 跨机器通信 | 适用 | 不适用 |

## 2.4.6 保护

> [!summary] 核心职责
> 确保系统资源不被非法访问和滥用。

**保护的泛化范围**

- **历史局限**：早期只有多用户、多道程序的大型计算机才需要资源保护。
- **现状普及**：随着**网络和互联网**的普及，保护已成为**所有计算机系统**的必选项。

**核心权限管理接口**

- **基础权限**：`set_permission()` 和 `get_permission()` 用于设置或获取文件、磁盘等资源的读写执行权限。
- **精细控制**：`allow_user()` 和 `deny_user()` 允许精准地允许或拒绝**特定用户**对特定资源的访问。
