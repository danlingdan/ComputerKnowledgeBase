## 3.1 进程概念

> [!summary] 核心定义
> 在不同类型的系统中，CPU 执行的活动有不同的叫法——批处理系统中称为**作业（Job）**，分时系统中称为**用户程序（User Program）**或**任务（Task）**。为了统一描述所有 CPU 执行的实体，引入了**进程（Process）**这一通用术语。

### 3.1.1 进程

> [!summary] 程序 vs 进程
> **程序（Program）**是被动实体，仅仅是存储在磁盘上的可执行文件（代码）；**进程（Process）**是活动实体，当程序被加载到内存并开始执行时，它就变成了进程。

**进程在内存中的组成部分**

![进程内存布局](assets/image-20260707030639405.png)

| 组成部分 | 功能描述 |
| --- | --- |
| **文本段/代码段（Text Section）** | 程序的可执行代码。 |
| **程序计数器 & 寄存器** | 记录当前运行的状态和上下文。 |
| **数据段（Data Section）** | 存储全局变量。 |
| **堆栈（Stack）** | 存储临时数据，如函数参数、局部变量和返回地址。 |
| **堆（Heap）** | 进程在运行时动态分配的内存区域。 |

**一对多的关系（进程 vs 程序）**

> [!example] 多实例运行
> 同一个程序可以对应**多个独立的进程**。例如，同时打开多个浏览器窗口或邮件客户端。它们**共享相同的文本段（代码）**，但拥有**各自独立的数据段、堆栈和堆**，互不干扰。

### 3.1.2 进程状态

> [!summary] 五种基本状态
> 进程生命周期经历五个阶段：新的、就绪、运行、等待、终止。

![进程状态转换](assets/image-20260707030854688.png)

- **新的（New）**：进程刚被创建，但尚未被操作系统接纳执行。
- **就绪（Ready）**：进程已经准备好，正在等待分配 CPU（处理器）资源。
- **运行（Running）**：进程的指令正在 CPU 上实际执行。
- **等待（Waiting）**：进程正在等待某个外部事件发生（如等待键盘输入或磁盘 I/O 完成），此时进程让出 CPU。
- **终止（Terminated）**：进程已经完成全部执行，系统准备释放其占用的资源。

### 3.1.3 进程控制块

> [!summary] 一句话定义
> **进程控制块（PCB）**是操作系统内核内部用来**表示一个进程**的唯一数据结构。每一个进程在系统中都有一个对应的 PCB，在一些操作系统中也被称为**任务控制块（TCB）**。

**PCB 中存储的关键信息（七大类）**

| 信息类别 | 具体内容 |
| --- | --- |
| **进程状态** | 当前进程处于什么阶段（新建、就绪、运行、等待、终止）。 |
| **程序计数器** | 指向该进程下一条需要执行的指令地址。 |
| **CPU 寄存器** | 包含累加器、堆栈指针、通用寄存器等。这是**进程切换（上下文切换）**的核心——发生中断时，操作系统会将当前进程的这些寄存器状态存入 PCB，以便后来恢复执行。 |
| **CPU 调度信息** | 如进程的优先级、调度队列的指针等。 |
| **内存管理信息** | 如基地址、界限寄存器、页表/段表信息等。 |
| **记账信息** | 记录该进程的 CPU 耗时、实际运行时间等，用于系统统计和资源计费。 |
| **I/O 状态信息** | 记录该进程当前占用的 I/O 设备列表以及已打开的文件列表。 |

### 3.1.4 线程

> [!summary] 从单线程到多线程
> 传统进程模型每个进程只有一个执行流（单线程），现代操作系统扩展为多线程，允许在一个进程内部拥有多个执行线程。

**传统单线程进程的局限性**

> [!problem] 串行执行的瓶颈
> 在单线程模型下，进程一次只能执行一个任务。例如，使用字处理软件时，无法在输入文字的同时让软件在后台进行拼写检查。

**现代多线程（Multithreading）的优势**

> [!solution] 并行执行的价值
> 多线程在**多核 CPU** 架构中极具优势，因为不同的线程可以真正被分配到不同的核心上**并行运行**，从而极大提升系统的执行效率。

**底层数据结构的变更**

> [!note] 后续章节预告
> 引入线程不仅改变了进程的模型，还使得操作系统内核的基础数据结构**进程控制块（PCB）** 发生了扩展，需要包含**每一个线程**的状态信息。第 4 章将对此进行详细探讨。

## 3.2 进程调度

> [!summary] 核心职责
> 进程调度决定何时将 CPU 分配给哪个进程，是操作系统实现多任务的关键机制。

### 3.2.1 调度队列

> [!summary] 三种核心调度队列
> 操作系统维护三种队列来管理进程：作业队列、就绪队列和设备队列。

**三种核心调度队列（Queue）**

| 队列类型 | 包含内容 |
| --- | --- |
| **作业队列（Job Queue）** | 系统中所有进入的进程。 |
| **就绪队列（Ready Queue）** | 驻留在内存中、已准备好运行、正在等待分配 CPU 的所有进程。 |
| **设备队列（Device Queue）** | 因等待特定 I/O 设备（如磁盘、打印机）而处于等待状态的进程列表。每个物理设备都有自己独立的队列。 |

**队列的底层实现**

> [!mechanism] 链表实现
> 这些调度队列通常通过**链表**来实现，以支持动态增加和删除进程。链表的**头节点**保存指向第一个和最后一个 **PCB（进程控制块）** 的指针；每个具体的 **PCB** 内包含一个指向队列中"下一个 PCB"的指针。

![调度队列结构](assets/image-20260707032130258.png)

**进程调度的生命周期流转（核心重点）**

> [!mechanism] 状态流转流程
> 1. **启动**：新进程首先进入**就绪队列**。
> 2. **执行**：调度器**分派**进程，它获得 CPU 开始执行。
> 3. **等待/切换事件**：运行期间可能发生三种情况：
>    - **I/O 请求**：进程主动发起 I/O，从 CPU 移出，进入**设备队列**（等待状态）。
>    - **创建子进程**：进程等待子进程执行完毕，进入**设备队列/等待队列**（等待状态）。
>    - **中断**：时间片用完或被抢占，进程被动释放 CPU，**回到就绪队列**。
> 4. **恢复循环**：当 I/O 完成或子进程结束，进程从"等待状态"移回**就绪队列**，等待下一次被分派。
> 5. **结束**：当进程完成全部任务后**终止**，操作系统将其从所有队列中删除，释放其 PCB 及占用的系统资源。

![队列图](assets/image-20260707032152342.png)

> [!tip] 队列图解读
> 在队列图中，**矩形**代表不同类型的队列，**圆圈**代表硬件资源（如 CPU、磁盘控制器），**箭头**则清晰地指示了进程在各个状态和队列之间的流转方向。

### 3.2.2 调度程序

> [!summary] 三种调度程序的分工
> 长期调度程序（作业调度）、短期调度程序（CPU调度）、中期调度程序（交换）各自承担不同层次的调度职责。

**三种调度程序的定义与分工**

| 调度程序 | 职责 | 执行频率 |
| --- | --- | --- |
| **长期调度程序（作业调度程序）** | 决定将哪些进程从磁盘的缓冲池加载到内存中执行。控制系统的**多道程序程度**。 | 极低（几分钟一次） |
| **短期调度程序（CPU 调度程序）** | 决定在已经准备好执行的进程中，将 CPU 分配给哪一个进程。 | 极高（每 100ms 甚至更短） |
| **中期调度程序** | 通过**交换（Swapping）**技术，将进程从内存换出到磁盘以释放内存，之后再根据情况换入回内存继续执行。 | 中等 |

**长/短期调度程序的性能差异**

> [!warning] 调度开销的重要性
> 如果调度本身花费 10ms，而进程运行 100ms，那么光调度就会消耗约 **9%** 的 CPU 时间。因此短期调度算法必须极其快速和高效。

**长期调度的关键策略：进程组合**

> [!strategy] I/O 密集型与 CPU 密集型的平衡
> 长期调度程序需要将 **I/O 密集型进程**（频繁等待 I/O，占用 CPU 少）和 **CPU 密集型进程**（需要大量 CPU 计算）进行**合理组合**。如果全是 I/O 密集型进程，CPU 会频繁闲置；如果全是 CPU 密集型进程，I/O 设备会闲置。

### 3.2.3 上下文切换

> [!summary] 一句话定义
> **上下文切换**是指当发生中断时，操作系统将 CPU 从当前运行的进程切换到另一个进程的过程：**保存（Save）**当前进程的状态（存储在它的 **PCB** 中），并**恢复（Restore）**即将运行的进程的状态。

**性能开销（纯开销）**

> [!warning] 纯开销特性
> 上下文切换消耗的 CPU 时间是**纯粹的开销（Overhead）**。因为在此期间，CPU 没有执行任何实际的用户程序或工作任务。切换耗时的典型值在**几毫秒**，具体取决于硬件内存速度、需要复制的寄存器数量等因素。

**硬件支持对效率的影响**

> [!example] 多组寄存器的优势
> 有些处理器（如 **Sun UltraSPARC**）在硬件上提供**多组寄存器**，此时切换只需简单地改变**寄存器组指针**即可，速度极快。但如果同时运行的进程数超过了硬件支持的寄存器组数量，系统依然只能退回到常规的内存复制模式。

**与内存管理的关联**

> [!note] 第8章预告
> 在拥有现代**虚拟内存**技术的系统中，上下文切换变得更加复杂。不仅需要保存和恢复 CPU 寄存器，还需要处理进程的**地址空间**。这引出了后续第 8 章要深入讨论的核心机制。

## 3.3 进程运行

> [!summary] 进程的生命周期
> 进程从创建到终止的完整生命周期，包括进程创建、进程终止以及特殊的僵尸进程和孤儿进程状态。

### 3.3.1 进程创建

> [!summary] 父子关系与进程树
> 进程由其他进程创建，创建者称为**父进程**，被创建的称为**子进程**。这种父子关系构成了树状的**进程树**。

**基础概念**

- **进程标识符（PID）**：每个进程拥有一个唯一的整型 PID，用于系统识别。以 Linux 为例，`init` 进程（PID=1）是所有用户进程的祖先。使用 `ps` 命令可查看系统的进程树结构。
- **资源分配**：子进程需要的资源（CPU、内存、文件等）可以从操作系统获取，或者从父进程处继承资源子集。合理限制资源可以防止系统超载。

### 3.3.2 进程终止

> [!summary] 两种终止方式
> 进程可以主动终止（`exit()`）或被动终止（父进程强制终止）。

**进程终止的方式**

| 终止方式 | 触发方式 | 返回值 |
| --- | --- | --- |
| **主动终止** | 进程执行完最后一条语句，通过系统调用 `exit()` 自行结束 | 可返回一个整数退出状态给父进程 |
| **被动/强制终止** | 父进程通过系统调用（如 Windows 的 `TerminateProcess()`）强制终止某个子进程 | 无 |

**父进程终止子进程的原因**

- 子进程消耗了超过分配的资源。
- 子进程原先肩负的任务已经完成或不再需要。
- 父进程即将退出，且系统禁止无父进程的子进程继续运行。这会产生**级联终止（Cascade Termination）**，即父进程终止时，所有子进程也会被系统一并终止。

**两种特殊的进程状态**

> [!warning] 僵尸进程（Zombie Process）
> 子进程已经终止（资源已释放），但其在**进程表（Process Table）**中的条目依然存在，原因是父进程尚未调用 `wait()` 来获取其退出状态。这是一个短暂的状态，一旦父进程执行 `wait()`，僵尸进程就会被彻底清理。

> [!warning] 孤儿进程（Orphan Process）
> 父进程在没有调用 `wait()` 的情况下提前终止，导致子进程变成"孤儿"。在 Linux/UNIX 系统中，此类进程会被系统根进程 **`init`**（PID 为 1）自动领养。`init` 进程会周期性地调用 `wait()` 来收集这些孤儿进程的退出状态并彻底回收资源，防止系统进程表溢出。

## 3.4 进程间通信

> [!summary] IPC 的必要性
> 协作进程需要通过进程间通信（IPC）机制交换信息，主要有共享内存和消息传递两种基本模型。

**进程的独立与协作**

- **独立进程**：不能影响或被其他进程影响，不共享数据。
- **协作进程**：能影响或被其他进程影响，需要共享数据。

**为什么需要进程协作？**

> [!reason] 四大好处
> 进程协作能够带来：**信息共享**（如并发访问文件）、**计算加速**（利用多核并行拆分任务）、**模块化**（功能解耦）、以及**用户操作的方便性**（如一边听音乐一边编译代码）。

### 3.4.1 共享内存系统

> [!summary] 核心机制
> 在内存中划出一块供进程共享的区域，进程直接读写该区域进行通信。

**共享内存的运作本质**

> [!mechanism] 打破隔离
> 操作系统默认阻止进程间相互访问内存。共享内存要求相关进程**"自愿取消"**这个限制，建立一块共用的内存区域。OS 只管分配这片内存；**数据的格式、存放位置、以及并发写入的同步**，全部由通信的进程自己负责，操作系统不再干预数据传输过程。

**生产者-消费者模型**

> [!example] 经典并发范例
> **生产者**生成数据，**消费者**处理数据。解决这个问题的核心，是建立一个中间缓冲池（缓冲区）。这本质上是"客户机-服务器"架构（如 Web 服务器与浏览器）的抽象。

**缓冲区的两种类型**

| 类型 | 特点 | 行为 |
| --- | --- | --- |
| **无界缓冲区** | 大小理论上无限 | 消费者可能得等数据，但生产者永远不会等 |
| **有界缓冲区** | 大小固定 | **满则停**（生产者等待），**空则停**（消费者等待） |

**基于"循环数组"的有界缓冲区实现**

> [!mechanism] 双指针逻辑
> - **数据结构**：使用固定大小的数组模拟循环队列。
> - **双指针**：`in` 指向下一个空位，`out` 指向下一个满位。
> - **判空**：`in == out`
> - **判满**：`(in + 1) % BUFFER_SIZE == out`（这种实现会**浪费一个存储空间**来区分空和满）。

**还未解决的核心难题：同步（重点提示）**

![共享内存同步问题](assets/image-20260707040515553.png)

> [!warning] 并发竞争风险
> 文中的代码使用了"忙等待"（`while` 空循环）来实现简单的同步。**致命缺陷**：这种方式并没有真正解决**并发竞争**问题。如果两个进程（或多个线程）同时读写 `in` 或 `out` 指针，或者同时写入缓冲区，数据就会混乱。

### 3.4.2 消息传递系统

> [!summary] 核心机制
> 进程通过操作系统内核交换消息包，不需要通信的进程共享相同的地址空间。

**什么是消息传递？**

> [!mechanism] 内核介入
> 操作系统（而非程序员）接管了数据交换的控制，提供 `send(message)` 和 `receive(message)` 两个基础操作。**核心优势**：特别适用于**分布式环境**（即跨越网络连接的不同计算机之间的通信，如互联网聊天程序）。

**消息类型设计的权衡**

> [!comparison] 定长 vs 变长消息
> - **定长消息**：系统底层实现简单，但给上层编程带来了限制。
> - **变长消息**：系统底层实现更复杂，但极大地方便了应用程序员的编程。

**通信链路的逻辑特性**

进程 P 和 Q 之间的通信，必须通过某种"通信链路（Communication Link）"，需要关注三个核心维度：
- **直接或间接的通信**（直接指明对方进程，还是通过中间实体如"邮箱"来投递）。
- **同步或异步的通信**（发送方是阻塞等待对方接收，还是发送完就继续执行；接收方如何等待消息）。
- **自动或显式的缓冲**（消息发送后，是自动暂存在系统缓冲区，还是需要程序员显式管理）。

#### 3.4.2.1 命名

**直接通信（Direct Communication）**

> [!mechanism] 硬编码 PID
> 发送者和接收者必须在系统调用中**明确指定对方进程的标识符（PID）**。例如 `send(P, msg)` 指定发给进程 P，`receive(Q, msg)` 指定从进程 Q 接收。

> [!problem] 硬编码的致命缺陷
> 无论是对称寻址还是非对称寻址，这种直接指明 PID 的方式都属于**"硬编码"**。一旦某个进程的 PID 因系统变更而改变，所有与其通信的其他进程代码都需要逐一修改，**模块化和灵活性极差**。

**间接通信（Indirect Communication）**

> [!mechanism] 邮箱解耦
> 通过一个中间抽象对象——**邮箱（Mailbox）**或**端口（Port）**来进行通信。发送方和接收方只需要指定邮箱的标识符，而不需要知道对方的进程 ID。这像现实生活中的"信箱"一样实现了**发送者和接收者的解耦**。

**多进程邮箱与消息分配**

> [!strategy] 三种处理方案
> 如果多个接收进程同时监听同一个邮箱，系统可以：
> 1. 限制一个链路只能关联两个进程。
> 2. 限制一次只能有一个进程执行 `receive()` 操作。
> 3. 由**操作系统按特定算法（如轮转法）随机挑选**其中一个进程接收，或由发送方指定接收者。

**邮箱的两种所有权归属**

| 类型 | 归属 | 特性 |
| --- | --- | --- |
| **进程拥有的邮箱** | 作为进程地址空间的一部分 | 进程是所有者（拥有接收权限）；其他进程是使用者（只能发送）；进程终止时邮箱随之消失 |
| **操作系统拥有的邮箱** | 系统全局资源 | 独立于进程；所有者可将接收特权转移给其他进程 |

#### 3.4.2.2 同步

**消息传递的四种同步行为**

> [!comparison] 阻塞 vs 非阻塞
> | 操作 | 阻塞 | 非阻塞 |
> | --- | --- | --- |
> | **发送** | 必须等待直到消息被成功接收 | 发出消息后立即继续执行 |
> | **接收** | 没有消息时挂起阻塞 | 无论是否有消息都会立即返回 |

**"交会"（Rendezvous）同步**

> [!mechanism] 对称汇合
> 当**双方都采用阻塞通信**（即 `send` 和 `receive` 均为阻塞）时，发送方和接收方就形成了**"交会（Rendezvous）"**。发送者和接收者会在通信时碰头，严格保持同步。

> [!solution] 简化生产者-消费者问题
> 在共享内存模型中，需要依靠复杂的"忙等待"或信号量来管理缓冲区边界。但在消息传递模型中，采用**"阻塞发送 + 阻塞接收"**的策略，**大幅简化了实现逻辑**。

#### 3.4.2.3 缓存

**缓冲的本质**

> [!mechanism] 临时队列
> 通信进程之间交换的消息，在实际传递之前会先驻留在操作系统维护的**临时队列**（或称通信链路）中。根据队列容量的不同，系统表现出不同的同步特性。

**三种缓冲机制对比**

| 类型 | 容量 | 发送方行为 | 接收方行为 |
| --- | --- | --- | --- |
| **零容量（零缓冲 / 同步发送）** | 0 | 必须阻塞直到接收者完全接收 | 必须阻塞直到有消息到达 |
| **有限容量（有界缓冲）** | 固定为 _n_ | 队列未满时非阻塞，已满时阻塞 | 队列非空时非阻塞，已空时阻塞 |
| **无限容量（无界缓冲）** | 理论上无限 | 从不阻塞 | 无消息时阻塞 |

> [!note] 系统分类术语
> - **无缓冲的消息系统（Zero-capacity）**：对应零容量。
> - **自动缓冲的消息系统（Automatic buffering）**：对应有限容量和无限容量。

## 3.6 客户机/服务器通信

> [!summary] C/S 架构的通信机制
> 客户机/服务器架构是分布式系统的核心模式，主要通过套接字、远程过程调用（RPC）和管道三种机制实现通信。

### 3.6.1 套接字

> [!summary] 基本概念
> 套接字由 **IP 地址**和**端口号**组成，是网络通信的"端点"。每个 TCP 连接都由一对独一无二的套接字（源IP:源端口 + 目的IP:目的端口）来唯一标识。

**端口规则**

- **小于 1024** 的端口通常被分配给众所周知的标准化服务（如 HTTP:80, FTP:21, Telnet:23）。
- 客户端发起连接请求时，操作系统会为其分配一个**大于 1024** 的临时端口。

**Java 套接字 API 的类型**

| 类型 | 类名 | 特性 |
| --- | --- | --- |
| **TCP 套接字** | `java.net.Socket` | 面向连接的可靠传输 |
| **UDP 套接字** | `java.net.DatagramSocket` | 无连接、不可靠但高效的传输 |
| **多播套接字** | `java.net.MulticastSocket` | 允许一份数据发送给多个接收者 |

**套接字的局限性**

> [!warning] 低级通信形式
> 套接字通信属于**低级的通信形式**。它本质上只是传递**"无结构的字节流"**。如果需要在进程间传递结构化数据（如对象、复杂记录），应用程序员必须自己编写序列化和反序列化的代码。

### 3.6.2 远程过程调用

> [!summary] 核心目标
> RPC 旨在让网络上的**远程过程调用**，对调用者来说感觉和**调用本地过程**一样。

**抽象与通信机制**

> [!mechanism] C/S 架构 + 消息传递
> RPC 采用**客户机-服务器**模型，建立在**消息传递**（IPC）之上。消息具有明确的结构（包含函数标识符和参数），而非无结构的数据包。

**存根（Stub）与参数封装（Marshaling）**

> [!mechanism] 透明调用流程
> 1. 客户端调用本地存根。
> 2. 客户端存根**封装（Marshaling）**参数（打包成网络可传输格式）。
> 3. 向服务器发送消息。
> 4. 服务端存根接收并解包，调用实际的服务例程。
> 5. 服务端将结果封装并发回，客户端存根接收并返回给调用者。

> [!tip] Windows 实现
> 在 Windows 系统中，使用 **MIDL（微软接口定义语言）** 来定义接口并自动生成存根代码。

**RPC 面临的三大核心挑战及解决方案**

| 挑战 | 解决方案 |
| --- | --- |
| **数据表示（大端 vs 小端问题）** | 引入**外部数据表示（XDR）**，客户端将数据转为机器无关的 XDR 格式再发送 |
| **调用语义（网络错误与可靠性）** | 采用"**最多一次（At most once）**"语义（通过时间戳去重）或"**正好一次（Exactly once）**"语义（需要额外的 ACK 确认） |
| **绑定（如何确定服务器的端口）** | 方案A：固定端口；方案B：动态绑定，通过"**交会服务程序 / 月老（Matchmaker）**"获取端口 |

**RPC 的实际应用（分布式文件系统）**

> [!example] DFS 实现
> RPC 机制是构建**分布式文件系统（DFS）**的基础。客户端通过 RPC 消息向服务器发送文件操作请求（`read`、`write`、`rename`、`delete` 等），服务端执行实际磁盘操作，将结果通过 RPC 返回给客户端。

### 3.6.3 管道

> [!summary] 基本定义
> 管道是 UNIX 系统最早使用的 IPC 机制。它为进程通信提供了一种**相对简单**的方法，但也存在**一定的局限性**。

**管道设计的四个属性**

- **方向性**：数据流是**单向**还是**双向**的？
- **并发传输**：如果支持双向，是**半双工**还是**全双工**？
- **进程关系**：通信的双方是否必须存在特定的关系（比如**父子进程**，或者是同一用户的进程）？
- **通信范围**：管道是**仅限于本地主机**使用，还是可以跨**网络**进行通信？

#### 3.6.3.1 普通管道

**基本特性（硬性限制）**

> [!warning] 三大限制
> - **单向通信（半双工）**：只能一端写入，另一端读取。若需双向通信，必须创建两个管道。
> - **通信范围：仅限本地**：只能在**同一台计算机**上进行，不能跨越网络。
> - **进程关系限制**：通信双方必须具有**亲缘关系（父-子进程）**。通常由父进程创建管道，并通过 `fork()`（UNIX）或 `CreateProcess()`（Windows）让子进程继承管道的文件描述符或句柄。

**Windows 系统的实现**

> [!mechanism] 匿名管道
> - **名称**：在 Windows 中被称为**匿名管道（anonymous pipe）**。
> - **创建**：使用 Windows API 函数 `CreatePipe()`。需要明确传递**读句柄**和**写句柄**。
> - **继承设置**：与 UNIX 自动继承不同，Windows 需要程序员显式设置继承属性。需要配置 `SECURITY_ATTRIBUTES` 结构允许句柄继承，并在 `CreateProcess()` 创建子进程时将子进程的**标准输入/输出重定向**到管道的对应端。

#### 3.6.3.2 命名管道

**命名管道 vs 普通管道的核心提升**

| 特性 | 普通管道 | 命名管道 |
| --- | --- | --- |
| **持久性** | 随进程消亡而消失 | 作为文件系统对象持久存在，直到被显式删除 |
| **关系限制** | 必须具有亲缘关系 | 无此限制，允许任意进程访问 |
| **双向性** | 单向（半双工） | 支持双向通信 |

**UNIX 的实现：FIFO**

> [!mechanism] 特殊文件
> - **创建与使用**：通过系统调用 `mkfifo()` 创建，在文件系统中表现为一个特殊文件。使用标准的 `open()`、`read()`、`write()`、`close()` 进行操作。
> - **局限性**：仅支持**半双工**；仅限于同一台机器内的进程通信；只支持**字节流**数据传输。
